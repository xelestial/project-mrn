# Redis Runtime UI Playtest Lessons

Date: 2026-05-01

## Authoritative Store Boundary

- In Redis-backed mode, Redis is the authoritative hot-state store. Process-local dictionaries are caches only.
- If a Redis-backed runtime status key is missing, backend readers must not fall back to stale process memory. Missing Redis runtime state means idle/absent state unless a live task in the same process proves otherwise.
- Store-backed session reads must refresh from the shared session store before returning public session state. Otherwise an API process can keep showing a deleted or archived session after the archive worker has already removed the Redis hot keys.
- Store-backed session creation must refresh before persisting. A stale process that creates a new session after cleanup can otherwise resurrect deleted sessions because the whole session set is persisted together.

## Worker And API Ownership

- Redis timing was not the root cause of the archived-session regression. The bug lived in backend ownership boundaries: API and worker processes were allowed to use their own old memory after another backend process had changed the shared state.
- Long-lived workers may cache for performance, but every active-session scan and every public read path must re-check the shared store when Redis persistence is enabled.
- Archive cleanup must be tested against at least two backend service instances or an equivalent stale-reader harness, not just the worker process that performed the cleanup.

## Screen-Visible Game Rules

- Rule events are not complete unless the player can see the effect on screen. `운수`, `잔꾀`, and `날씨` effects that mutate public state must surface as readable overlay/reveal items, not only as later board changes.
- UI verification must check causal readability, not just final state. A test is incomplete if it only proves that cash, shards, hand count, or position eventually changed; it must also prove that the screen explains why the change happened at the moment of the effect.
- Mandatory watch targets for every browser playtest are `인물 드래프트`, `지목`, `지목효과 발동`, `내 턴 아닐 때 액션`, and `주사위 던지기`. A test report must say which were observed live, which were covered by targeted e2e/regression tests, and which were not reached.
- `잔꾀` use must visibly remove the used card from the hand/tray immediately, update public/hidden hand counts, and show the effect label/result before the next decision can visually dominate the screen.
- `운수` results that change money, shards, movement, or bankruptcy risk must leave a readable reveal/feed item long enough for the player to connect the card to the resource delta. A later cash total alone is not acceptable evidence.
- Character passive bonuses such as `객주` lap reward enhancement need their own visible breakdown or a combined reward line that shows base reward plus bonus. Applying the bonus silently in backend state makes the rule feel invisible even when the math is correct.
- Weather must remain visible as current round context during decisions and effect resolution. If weather changes resource or marker behavior, the UI needs a named weather line rather than only a generic round/turn header.
- Public hidden-card counts must be derived from an actual hidden card identity. If `hidden_trick_deck_index` is absent or no longer matches a card in `trick_hand`, selectors must expose `hidden_trick_count: 0`; otherwise the UI can claim a hidden trick exists while every card is public.
- Backend `view_state.reveals` is authoritative for the frontend when present. It must include every high-signal public effect that the fallback selector would otherwise show: `weather_reveal`, `trick_used`, `fortune_resolved`, `lap_reward_chosen`, `bankruptcy`, and `game_end`.
- `game_end` needs both a stream event and an explicit visual overlay/result state. A finished backend status alone is not enough for screen verification.
- Labels must communicate hard rule constraints. Lap reward copy now says the budget must be spent exactly, because a disabled confirm button at `9/10` looked like a UI bug without that rule text.

- A later 2H+2AI continuation reached round 2 after AI P3/P4 auto-play, proving Redis kept authoritative player positions and trick counts across humans and AIs. `current_state` and `view_state:player:*` agreed on P1/P2/P3/P4 pawn positions and hand counts.

A reroll-capable card such as `뭔칙휜` may create additional movement decisions, but the visible prompt count must be bounded by the rule and older movement prompts must be superseded when a newer same-player/same-turn movement prompt is issued.
- The final P1 screen also showed a consumed `뭔칙휜` in the bottom trick tray while Redis and the player panel both reported P1 `trick_hand: 0` / `Trick0`. State storage was correct; the frontend projection/render path still allowed stale hand UI to survive prompt replay.

Redis correctly preserved and replayed that bad command; it was not a Redis timing bug. Timeout fallback must pick an explicit default or the first legal prompt choice, and test coverage must assert movement fallback records a legal choice such as `dice`.
- A near-frozen P1 round 2 screen was caused by command wakeup replaying stale `decision_resolved` commands that did not match the current `waiting_prompt_request_id`. Redis was only storing the command stream; the backend wakeup worker must skip mismatched waiting-input commands and advance its consumer offset instead of rehydrating the same prompt over and over.
- A hidden-trick selection regression looked like a UI hand-count problem, but the root cause was a mid-turn continuation boundary. `hidden_trick_card` is emitted after the used trick has already mutated state; resuming from that prompt must enqueue a continuation action instead of replaying the whole turn from the start. Redis `current_state` correctly held `hidden_trick_deck_index`, but the selector also had to consume direct `trick_window_closed` hand fields (`public_tricks`, `hidden_trick_count`, `hand_count`) so projected `view_state` matched the authoritative state.
- Docker compose server/worker containers are image-based and do not mount backend source. After a backend fix, a local green unit test is not enough for browser validation; rebuild and restart `server`, `prompt-timeout-worker`, and `command-wakeup-worker` before judging the screen.
- In the fresh 2-human + 2-AI retest, session `sess_CrAt2zEMf9W79JjFauDvHDf7` no longer showed the stale-command event flood. After P1 accepted the blocking burden exchange, AI turns advanced to round 2, and the stream stayed stable at the active human purchase prompt (`latest_seq: 206`) after repeated waits.
- The same retest showed why the screen can still feel frozen: P4's AI `아주 큰 화목 난로` caused a P1 burden exchange before the source trick/effect was visible to P1, and the round 2 weather removed P1 burden cards while dropping cash from 18 to 10 before the next trick prompt without a durable weather/result explanation in the decision area. Backend state was correct; causal readability was not.
- Weather and AI-triggered side effects must be treated as prompt blockers with mandatory cause context. If an effect creates an immediate human follow-up prompt (`burden_exchange`, purchase, reward choice, etc.), that prompt must include or sit beside the source effect name and resource delta so the player does not interpret the screen as stalled.
- Supply/burden exchange replay must freeze the set of burden cards that were eligible at the start of the supply action. Tracking only `processed_burden_deck_indices` is not enough: if another player's prompt pauses the action and the turn later replays, newly drawn burdens can be pulled into the same threshold chain and send the UI back to an earlier player's burden prompt. Redis preserves the replay command faithfully; the backend action payload must carry `eligible_burden_deck_indices_by_player`.
- A Docker or host memory failure can mimic a frontend freeze. If the page reports `ERR_CONNECTION_REFUSED` for `9090` and `docker compose ps` cannot connect to `/Users/sil/.docker/run/docker.sock`, treat it as runtime infrastructure loss first, not a game-rule regression.
- A post-restart frozen-looking P1 screen was caused by `/runtime-status` returning the public recovery `view_state` even when the caller supplied a valid seat token. On reload, the frontend seeded itself from a recovery payload with no private active prompt or hand tray, while Redis stream history still contained the correct P1 `trick_to_use` prompt. Recovery endpoints must rebuild `view_state` for the authenticated `ViewerContext` from stream history rather than trusting a generic/public cached projection.
- Freshness metadata can be correct for the wrong viewer. A projected view cache at the latest sequence is not sufficient if it was produced for public recovery and later served to a seat. Viewer-specific recovery should either key strictly by viewer or force a rebuild before hydrating a reconnecting player.
- The 1-human + 3-AI full-game replay exposed a different frozen-looking reload path: `/replay` projected each event by re-reading the full Redis stream and rebuilding view state for that prefix. A finished game with 881 stream entries made replay slow enough to block follow-up recovery reads, so the browser sat on board hydration. Replay export must take one stream snapshot, redact messages in memory, and attach the latest view state once instead of doing per-event full-history projection.
- A finished-session reload now restores the board and players, but the screen still lacks a strong game-end state; it can show the final board with a stale-looking actor headline such as `P2's turn`. Full-game screen verification should separately assert a visible `game_end` or results affordance, not only `runtime.status == finished`.
- Memory-aware browser QA must use one browser context per checkpoint and close it immediately. Long playtests should rely on API polling between human prompts, not many persistent tabs, and the dev server/test browser processes must be stopped at the end of the run.
- A `긴장감 조성` play report exposed a stale prompt lifecycle gap, not a card-effect rollback. The card correctly queued `resolve_trick_tile_rent_modifier`, but selectors did not retire the prior `trick_to_use` prompt on `trick_used`, so the UI could briefly fall back to the old selection surface before the follow-up `trick_tile_target` prompt arrived. Any prompt that is completed by a domain event must have a phase-progress close rule in both backend `view_state` and frontend fallback selectors; acknowledgements alone are not enough during replay/reconnect gaps. Add a browser click regression for these cases, with a deliberate delay before the follow-up prompt, so stale prompt resurrection is visible.
- A `극심한 분리불안` play report exposed a turn-continuation gap inside the priority action queue. The queued movement and arrival actions executed in order, but the engine did not append a `continue_after_trick_phase` action after the trick-generated queue. When the queue drained, `turn_index` still pointed at the same player, so the next transition restarted turn start and could ask for another `trick_to_use`. Any phase that defers by queuing actions must also queue or persist its continuation token; "the queue ran correctly" is not the same as "the phase advanced."
- A `지목 -> 잔꾀 -> 지목 -> 잔꾀` loop exposed another replay-boundary case inside one turn-start transition. Mark-target character abilities run before `trick_to_use`, and movement runs after it; if a checkpoint is pending on `trick_to_use` or `movement`, replay must rewind far enough to consume the earlier `mark_target` decision with the original stable request id. Seeding only from `pending_prompt_instance_id - 1` or the normal trick+movement pair can resurrect the mark prompt and make accepted trick decisions look ignored.
- Draft order and turn order are separate contracts. Marker/draft order is valid for character selection and marker-context display, but once `round_order` exists it owns actual turn/player-strip ordering. Backend `view_state`, frontend raw fallback selectors, and React layout code must all respect the same precedence; a correct selector is not enough if the final component sorts seats by seat number again.
- Mark-target prompts must preserve explicit no-target choices after candidate filtering. Candidate ordering answers "which targets are legal and in what order"; it must not silently remove the player-controlled `none` choice.
- A pre-fix live `1 human + 3 AI` Redis browser session on 2026-05-01 reached round 2 without console errors, blank board, or player-card loss, but the e2e suite failed on effect/spectator selectors. Treat this as a real product risk: a screen can be visually playable while automated checks cannot prove that `운수`, `잔꾀`, `날씨`, rent/payoff, or worker fallback effects were causally readable.
- Stable test selectors are part of the UI contract for rule visibility. If the UI replaces `spectator-turn-panel`, `spectator-turn-weather`, `spectator-turn-worker`, or `board-event-reveal-*` with a new visual presentation, the new components must expose equivalent stable selectors before old tests are removed.
- `REDIS-UI-10` closed on 2026-05-01 only after `npm --prefix apps/web run e2e:human-runtime` passed all 18 checks and targeted browser parity checks passed for `긴장감 조성` and `극심한 분리불안`. The closure fix restored spectator/core-action/reveal DOM contracts instead of relying on a visually similar replacement.
- Cause/effect surfaces should be ordered by player-causal importance, not raw newest or mount order. Worker fallback, marker/flip, and fortune resolution must stay ahead of ambient context such as weather, character, `turn_start`, or a later unrelated rent event when the goal is explaining why the turn advanced.
- Character/draft prompts must be layout-tested as blocking game controls, not cosmetic modals. A prompt that requires body-level scrolling on desktop can hide choices, action buttons, or effect context and should fail the release gate.
- Event feed and reveal stacks must be both human-visible and machine-verifiable. If an effect is only visible in a transient animation or an unaddressable canvas/visual layer, browser QA cannot prove that the user can understand the cause of a resource or position change.
- Browser QA must always end by closing the browser context and stopping Docker services. Memory pressure from accumulated tabs can mimic game stalls and contaminate the next diagnosis.
- A finished engine transition is a terminal screen state even if the backend did not emit a dedicated `game_end` event. Session `sess_ym3kcRhwYMZzlSh0DJCWUZ2w` reached `final_checkpoint.status=finished` and ended with `engine_transition(status=finished, reason=end_rule)`, but the latest backend turn projection still described P1 `turn_end_snapshot` / `movement`. The UI looked stalled because selectors trusted that stale turn projection. Frontend selectors must promote terminal engine transitions to `game_end`, clear actor/prompt state, and surface a visible end banner/overlay.
- Test harnesses must not keep waiting once runtime/session state is terminal and there is no pending prompt or scheduled action. If the screen still shows a turn prompt/actor in that state, fail as a UI/runtime projection bug and document the exact stream tail.
- The parity suite exposed a UI layout failure in `quick start human vs ai enters match and surfaces the first human prompt`: hidden-trick choice cards no longer shared a single row and one card wrapped about 91px lower than the first row. The root cause was a CSS contract mismatch: the component emitted `prompt-overlay-hidden_trick_card`, while the hidden-trick layout rules targeted `prompt-overlay-hidden_trick`, so the six-slot trick grid never applied. Prompt CSS selectors must use the exact `requestType`-derived overlay class or a shared helper-generated class; shortened aliases silently remove critical control layout.
- Full web rule validation exposed a theater payoff regression: `buildPayoffSceneItems()` dropped same-turn economy payoff beats whenever a fortune beat existed, then rendered the remaining beats in newest-first order. This made a turn with `tile_purchased -> fortune_drawn -> fortune_resolved` appear as only `fortune_resolved -> fortune_drawn`. Effect surfaces must preserve the whole same-turn payoff chain in chronological order so players can connect purchase, reward/loss, fortune, trick, and weather causes to the final state.
- A live draft/turn-order regression exposed a checkpoint replay boundary at round start. `_start_new_round()` clears per-round draft state before running draft, final character choice, hidden trick setup, and round-order sorting in one transition. If a pending round-start prompt resumes from only `pending_prompt_instance_id - 1`, earlier draft prompts get skipped or re-numbered on replay, so the selected character, priority order, and character start abilities can all diverge from what the player chose. When `current_round_order` is still empty, prompt replay must restart from sequence `0` and consume every stored decision in that round-start transition in the original order.
- Stale `current_round_order` is also dangerous in the opposite direction. A checkpoint can still carry the previous round order while the active pending prompt belongs to the next round's `draft_card` or `final_character` flow. Round-start replay should key off pending prompt type/request id, not only an empty order list.
- This failure can look like several unrelated UI bugs: one-card draft choices appearing to consume turns out of priority order, final character selection returning to draft, and character abilities not firing. Treat those symptoms as one runtime replay integrity class before patching individual screens.
- Commit forensics narrowed this rule-regression cluster to `fd2e5e4` (`Fix Redis runtime playtest regressions`) as the first high-risk change, not because Redis itself was wrong, but because that commit changed engine continuation, prompt request-id replay, stale command wakeup, and recovery projection behavior together. When draft order, final character choice, hidden trick setup, and character-skill firing all look wrong at once, compare against the last known-good engine/runtime commit before treating it as a display-only bug.
- First-round setup effects must not eliminate a player before the first player choice. If weather is revealed before draft, initial trick-card dealing must happen after that first weather reveal or the starting hand can be punished by weather before players have selected characters.
- When draft or turn order looks globally broken, first assert `round_start.alive_player_ids` and the eventual `round_order` length. A missing player at round start can masquerade as draft, priority, targeting, and ability failures at the same time.
- Mark-target parsing must be tested with the real decision invocation args/kwargs. Rebuilding legal choices from an empty call context can make a visible legal target fail to resolve through the server decision gateway.
- Public mark-target labels are card-face guesses for future priority slots, not a guarantee that the visible final character name is the only legal label. Tests and UI copy should make this distinction clear enough that labels from a player's unchosen drafted card do not look like impossible targets.
- Mark effects are not ordinary same-turn flavor text. A queued mark from `만신`, `산적`, `추노꾼`, or `박수` resolves at the very beginning of the target player's turn, before `turn_start`, character-start abilities, trick prompts, movement, and landing effects. The engine should keep this as a scheduled `turn_start` action, and the UI must label `mark_queued` as a high-priority effect so it does not look like the current actor's turn suddenly restarted or the target turn skipped the mark payoff.
- A live `1 human + 3 AI` browser pass exposed a separate 4-player draft rule violation: production `GPT/engine.py` only asked for the first draft card, then randomly assigned every second draft card before final character selection. The screen could jump from P1's first draft choice straight toward turn actions, making selected characters and later character-skill behavior impossible to validate. Four-player draft must be asserted as phase 1 clockwise choices, phase 2 reverse choices, then final character confirmation before any trick/movement prompt.
- A marker owner can legitimately receive the final remaining card in phase 2 of the four-player snake draft, but that is a forced draft pick, not a human decision. The engine should auto-resolve any one-card draft pool, still emit the `draft_pick` event in snake order, and only prompt players when at least two draft choices exist.
- Marker card flips belong to the just-finished round's final turn end, not to the next round prelude. If `_start_new_round()` resolves `pending_marker_flip_owner_id`, the stream stamps `marker_flip` with the next `round_index` and `weather` phase, making the UI feel like the card flipped before weather/draft. Round-boundary code must resolve marker transfer and flip before incrementing the turn/round cursor, and reveal ordering must place `marker_transferred` before `marker_flip`.
- A later live session (`sess_J_kllISLYrI3hy9uc4ZUS4ty`) proved backend/Redis could be correct while the screen stayed stale: `/replay?token=...` exposed active `hidden_trick_card` at stream seq 78-80, but the browser still showed the earlier `#50 Draft pick` spectator state and no trick choices. The root cause was frontend recovery sequencing: synthetic replay projections could be assigned a sequence that the already-advanced client treated as old, so reducer logic kept the stale visible state. Replay projections must be sequenced beyond the caller's current client seq, and reducer recovery must preserve latest `view_state` projections even when they arrive through an old replay batch.
- The same failure also had a timing side: the first recovery reads after a decision can legitimately observe an incomplete backend transition. In the reproduced final-character click, early `/replay` calls returned only about 30 events while the backend later finished with the hidden-trick prompt at seq 78-80. Decision recovery must include a longer retry window, not only sub-2-second fetches, because prompt-producing transitions can be delayed by AI/draft replay work even when Redis is healthy.
- The measured final-character path for seed `2026050123` took roughly 10-11 seconds from accepted final-character decision to hidden-trick `prompt_required`. A browser harness that declares failure after 12 seconds can report a false stall just before the UI recovers. Keep live rule-check waits above the measured backend transition window, and treat the duration itself as a UX/performance signal to watch separately.
- Browser rule QA must compare the authenticated replay projection to the DOM whenever a prompt appears stalled. If `view_state.prompt.active` contains a prompt for the current seat and the matching control is absent from the screen, fail it as a projection/display bug even when Redis and backend state are correct.

- Do not treat every root `players` array as a full board snapshot. Lightweight setup payloads such as `initial_public_tricks` can contain only trick metadata, and letting them replace the live snapshot can blank active character faces or make draft state look corrupted.

## Playtest Rule

- A 4-human browser playtest must use four independent browser contexts/tokens and assert both stream legality and visible UI state.
- API-only completion is not a pass. The pass condition is: one full game reaches `game_end`, the board remains visible, and all public-effect families (`운수`, `잔꾀`, `날씨`) produce readable visible feedback during play.
- For faster regression loops, a 2-human + 2-AI session is acceptable only if it explicitly verifies immediate UI deltas for hand removal, shard gain, money loss/gain, weather context, and passive reward bonuses. It does not replace the full 4-human end-to-end pass.
- A quick `1 human + 3 AI` smoke test is useful for boot, Redis wiring, and prompt progression, but it is not enough to close effect-display regressions unless it also captures stable DOM evidence for reveal/event surfaces.
- Every playtest report must explicitly account for the mandatory watch targets: `인물 드래프트`, `지목`, `지목효과 발동`, `내 턴 아닐 때 액션`, and `주사위 던지기`. If live play does not naturally hit one, mark it as not reached and cover it with targeted e2e or regression tests before calling that surface verified.
