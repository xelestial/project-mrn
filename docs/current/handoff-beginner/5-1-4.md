호스트가 `POST /api/v1/rooms/{room_no}/start`를 호출하면 서버가 세션을 만들고, 세션을 시작하고, 런타임을 시작합니다. 반환값에는 `session_id`와 좌석별 `session_tokens`가 들어갑니다.

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 API 흐름 중 `방 시작`을 설명합니다. API를 읽을 때는 요청/응답 필드만 보지 말고, 그 호출 다음에 어떤 stream event와 화면 변화가 따라오는지 같이 봐야 합니다.

### 관련 코드와 문서

- `docs/current/api/online-game-api-spec.md`
- `apps/server/src/routes/`
- `apps/web/src/infra/http/sessionApi.ts`
- `apps/web/src/infra/ws/StreamClient.ts`
- `apps/web/src/hooks/useGameStream.ts`

### 실제 처리 흐름과 기대 값

- 이 문서의 API 흐름: 방 시작.
- 사용자 행동/요청: 호스트가 시작 버튼을 누르면 `startRoom`이 나갑니다.
- 서버 처리: `RoomService.start_room`으로 방을 잠그고 `SessionService.start_session`, `StreamService.publish`, `RuntimeService.start_runtime`으로 이어집니다.
- 기대 값/검증 포인트: `session_id`, 좌석별 `session_token`, `status=started/running`이 만들어지고 프론트가 match route와 WebSocket 연결로 넘어가야 합니다.

### 수정할 때 주의할 점

- REST는 시작/조회/복구에, WebSocket은 실시간 게임 진행에 더 가깝습니다.
- 모든 에러는 ok/data/error envelope 기준으로 읽으세요.
- API 응답과 이후 stream event를 한 흐름으로 기록하면 디버깅이 쉬워집니다.

### 확인 방법

- `.venv/bin/python -m pytest apps/server/tests/test_sessions_api.py apps/server/tests/test_stream_api.py -q`
- `npm --prefix apps/web run test -- src/infra/http --run`

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
