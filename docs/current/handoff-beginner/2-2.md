# 2-2. 핵심 위치

Status: ACTIVE

현재 엔진을 읽을 때의 우선순위는 `runtime_modules`의 세부 모듈이 먼저이고, `engine.py`는 그 모듈들이 아직 위임하는 기존 규칙 의미론을 확인할 때 봅니다. 즉, 이 프로젝트의 런타임은 모듈화되어 있지만 모든 규칙 코드가 작은 모듈 파일로 완전히 이관된 상태는 아닙니다.

- `GPT/runtime_modules/contracts.py`: Frame, Module, checkpoint, prompt continuation 계약
- `GPT/runtime_modules/runner.py`: 현재 활성 frame/module을 한 단계씩 진행하는 실행기
- `GPT/runtime_modules/round_modules.py`: 라운드 시작, 날씨, 드래프트, 턴 스케줄, 라운드 종료 모듈
- `GPT/runtime_modules/turn_modules.py`: 한 플레이어 턴 내부 단계의 모듈 정의
- `GPT/runtime_modules/sequence_modules.py`: 잔꾀, 이동, 도착, 구매, 후속 액션 같은 중첩 흐름
- `GPT/runtime_modules/simultaneous.py`: 보급처럼 여러 플레이어 결정이 모여야 하는 동시 처리
- `GPT/runtime_modules/prompts.py`: prompt resume/continuation 검증
- `GPT/engine.py`: `ModuleRunner`가 아직 호출하는 기존 규칙 helper, 이벤트, 상태 전이 의미론
- `GPT/test_runtime_*.py`, `GPT/test_rule_fixes.py`: 모듈 런타임과 규칙 회귀 테스트

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 엔진/런타임 관점의 `핵심 위치`을 설명합니다. 새 코드를 읽을 때는 `engine.py` 한 파일을 “엔진 전체”로 보지 말고, `contracts.py -> runner.py -> *_modules.py -> engine.py helper` 순서로 읽는 것이 안전합니다.

### 관련 코드와 문서

- `GPT/runtime_modules/contracts.py`
- `GPT/runtime_modules/runner.py`
- `GPT/runtime_modules/round_modules.py`
- `GPT/runtime_modules/turn_modules.py`
- `GPT/runtime_modules/sequence_modules.py`
- `GPT/runtime_modules/simultaneous.py`
- `GPT/engine.py` (완전한 출발점이 아니라 레거시 semantic kernel/bridge)
- `apps/server/src/services/runtime_service.py`
- `docs/current/runtime/end-to-end-contract.md`

### 실제 처리 흐름과 기대 값

- 이 문서의 엔진 흐름: 2-2. 핵심 위치.
- 시작 지점: 백엔드의 `RuntimeService.start_runtime`이 엔진과 모듈 runner를 만들고, prompt가 필요한 순간에는 런타임이 멈춥니다.
- 결정 재개: 사용자의 WebSocket decision은 `PromptService`에 저장되고 `DecisionGateway`가 같은 `request_id`를 소비한 뒤 해당 frame/module을 재개합니다.
- 기대 값/검증 포인트: checkpoint에는 `runtime_frame_stack`, `active_module_id`, prompt continuation 정보가 남아야 하고, 결과는 event와 `view_state`로만 프론트에 전달되어야 합니다.

### 수정할 때 주의할 점

- 새 런타임 흐름은 `runtime_modules`에 추가하고, `engine.py`에는 기존 helper를 호출하거나 점진 이관할 때만 손대세요.
- 런타임 단계가 꼬이면 frame/module/request_id를 같이 추적하세요.
- 프롬프트가 뜨는 단계에서는 엔진이 잠시 멈춘 것이 정상입니다.
- 자동 선택과 사람 선택이 같은 상태 전이 결과를 만들어야 합니다.

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
