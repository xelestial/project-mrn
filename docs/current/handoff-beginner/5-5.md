## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 API 흐름 중 `상태 복구 흐름`을 설명합니다. API를 읽을 때는 요청/응답 필드만 보지 말고, 그 호출 다음에 어떤 stream event와 화면 변화가 따라오는지 같이 봐야 합니다.

### 관련 코드와 문서

- `docs/current/api/online-game-api-spec.md`
- `apps/server/src/routes/`
- `apps/web/src/infra/http/sessionApi.ts`
- `apps/web/src/infra/ws/StreamClient.ts`
- `apps/web/src/hooks/useGameStream.ts`

### 실제 처리 흐름과 기대 값

- 이 문서의 흐름: 상태 복구 API.
- 사용자 행동/요청: 새로고침/끊김/늦은 접속에서 `runtime_status`, replay, WebSocket resume을 조합해 최신 상태를 되찾습니다.
- 서버 처리: 서버는 stream replay window와 런타임 checkpoint를 기준으로 복구 가능 여부를 판단합니다. gap이 너무 오래되면 최신 버퍼 또는 명시적 recovery 경로를 사용해야 합니다.
- 기대 값/검증 포인트: 기대 값은 마지막으로 처리한 `seq`, replay된 메시지 수, `runtime_status.checkpoint.runtime_frame_stack`의 active module입니다.

### 수정할 때 주의할 점

- REST는 시작/조회/복구에, WebSocket은 실시간 게임 진행에 더 가깝습니다.
- 모든 에러는 ok/data/error envelope 기준으로 읽으세요.
- API 응답과 이후 stream event를 한 흐름으로 기록하면 디버깅이 쉬워집니다.

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
