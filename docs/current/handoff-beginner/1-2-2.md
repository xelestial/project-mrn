4인 기준으로 캐릭터 카드를 4장 나누고, 마커 보유자부터 마커 방향대로 첫 번째 선택을 합니다. 다시 4장을 나눈 뒤에는 반대 순서로 두 번째 선택을 합니다. 최종적으로 각 플레이어는 후보 2장 중 1장을 골라 그 라운드의 캐릭터로 사용합니다.

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 게임 규칙 중 `캐릭터 드래프트`을 설명합니다. 규칙 문서는 UI 문구가 아니라 엔진이 실제로 따라야 하는 계약으로 읽어야 합니다. 화면에서 그럴듯하게 보여도 엔진 이벤트와 테스트가 다르면 규칙 구현이 맞다고 볼 수 없습니다.

### 관련 코드와 문서

- `docs/current/Game-Rules.md`
- `GPT/runtime_modules/round_modules.py`
- `GPT/runtime_modules/turn_modules.py`
- `GPT/engine.py` (모듈에서 아직 호출하는 레거시 규칙 helper 확인용)
- `GPT/test_rule_fixes.py`

### 실제 처리 흐름과 기대 값

- 이 문서의 규칙 흐름: 캐릭터 드래프트.
- 게임 안에서 시작되는 지점: 라운드 준비 중 엔진이 각 플레이어에게 캐릭터 후보 선택 prompt를 만들 때 시작됩니다.
- 처리 방식: 서버가 viewer별 prompt를 보내고, `PromptOverlay` 선택은 WebSocket decision으로 돌아와 `DecisionGateway`가 같은 `request_id`를 소비합니다.
- 기대 값/화면 결과: 본인에게만 후보/선택지가 보이고, `decision_ack.status=accepted` 뒤 최종 캐릭터 후보 상태가 view_state에 반영되어야 합니다.

### 수정할 때 주의할 점

- 규칙을 바꾸면 프론트 표시보다 엔진 테스트를 먼저 고치거나 추가하세요.
- 카드 이름 문자열보다 card id, trait, module metadata 같은 식별자를 우선 사용하세요.
- 즉시 효과인지, 대상 턴 시작에 해결되는 효과인지 구분하세요.

### 확인 방법

- `.venv/bin/python -m pytest GPT/test_rule_fixes.py -q`
- `.venv/bin/python -m pytest apps/server/tests/test_runtime_service.py -q`

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
