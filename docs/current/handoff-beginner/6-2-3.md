## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 `WebSocket 연결` 기능을 따라갑니다. `useGameStream`이 `StreamClient`를 만들고, 서버 `stream_ws`가 토큰을 검증한 뒤 `StreamService.subscribe()`로 연결합니다. 기능을 고칠 때는 한 계층만 보지 말고 엔진, 백엔드, 프론트 표시가 같은 흐름을 가리키는지 함께 확인해야 합니다.

### 관련 코드와 문서

- `apps/web/src/infra/ws/StreamClient.ts`
- `apps/web/src/hooks/useGameStream.ts`
- `apps/server/src/routes/stream.py`
- `apps/server/src/services/stream_service.py`

### 실제 처리 흐름과 기대 값

- 이 문서의 흐름: WebSocket 연결.
- 사용자 행동/트리거: 게임 화면에 들어오거나 새로고침 뒤 저장된 세션 토큰을 복원할 때 시작되는 흐름입니다.
- 프론트 경로: `App.tsx`가 `sessionId`와 `sessionToken`을 `useGameStream`에 넘기고, `StreamClient.connect`가 `/api/v1/sessions/{session_id}/stream?token=...`으로 연결합니다.
- 백엔드/엔진 경로: 서버는 `routes/stream.py`에서 토큰을 검증하고 `StreamService.subscribe`로 구독자를 등록합니다. 연결 직후 heartbeat와 replay/resume 응답이 같은 소켓으로 내려옵니다.
- 기대 값/화면 결과: 기대 값은 `connected=true`, 증가하는 `seq`, `heartbeat.payload.runner_kind`, 필요 시 `active_module_id`입니다. 화면은 연결 패널이 온라인 상태가 되고, stream selector가 최신 `view_state`를 반영해야 합니다.

### 수정할 때 주의할 점

- 상위 문서와 하위 문서를 함께 읽고, 관련 테스트를 먼저 찾으세요.

### 확인 방법

- `python3 tools/plan_policy_gate.py`
- 관련 서버/프론트/엔진 테스트를 함께 실행하세요.

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
