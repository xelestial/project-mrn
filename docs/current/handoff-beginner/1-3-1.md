# 1-3-1. 지목은 보통 플레이어가 아니라 캐릭터를 찍는다

문서 위치: [1. 게임 룰](1.md) > [1-3. 턴 시작 전 효과](1-3.md) > [1-3-1. 지목은 보통 플레이어가 아니라 캐릭터를 찍는다](1-3-1.md)

자객, 산적, 추노꾼, 박수, 만신 같은 효과는 "누구를 찍느냐"가 아니라 "어떤 캐릭터를 찍느냐"로 처리되는 경우가 많습니다. 실제 대상 플레이어는 그 캐릭터를 가진 플레이어로 해석됩니다.

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 게임 규칙 중 `지목은 보통 플레이어가 아니라 캐릭터를 찍는다`을 설명합니다. 규칙 문서는 UI 문구가 아니라 엔진이 실제로 따라야 하는 계약으로 읽어야 합니다. 화면에서 그럴듯하게 보여도 엔진 이벤트와 테스트가 다르면 규칙 구현이 맞다고 볼 수 없습니다.

### 관련 코드와 문서

- `docs/current/Game-Rules.md`
- `GPT/runtime_modules/turn_modules.py`
- `GPT/runtime_modules/sequence_modules.py`
- `GPT/runtime_modules/modifiers.py`
- `GPT/engine.py` (모듈에서 아직 호출하는 레거시 규칙 helper 확인용)
- `GPT/test_rule_fixes.py`

### 실제 처리 흐름과 기대 값

- 이 문서의 규칙 흐름: 지목 대상은 플레이어보다 캐릭터 의미가 우선입니다.
- 게임 안에서 시작되는 지점: 턴 시작 전 지목 prompt가 생성될 때입니다.
- 처리 방식: `selectMarkTargetCharacterSlots`가 선택 가능한 캐릭터 슬롯을 보여주고, decision payload는 대상 캐릭터/슬롯 의미로 엔진에 전달됩니다.
- 기대 값/화면 결과: 이벤트에는 지목한 캐릭터와 실제 영향을 받은 플레이어가 설명되어야 하며, 플레이어 번호만 보고 오판하면 안 됩니다.

### 수정할 때 주의할 점

- 규칙을 바꾸면 프론트 표시보다 엔진 테스트를 먼저 고치거나 추가하세요.
- 카드 이름 문자열보다 card id, trait, module metadata 같은 식별자를 우선 사용하세요.
- 즉시 효과인지, 대상 턴 시작에 해결되는 효과인지 구분하세요.

### 확인 방법

- `.venv/bin/python -m pytest GPT/test_rule_fixes.py -q`
- `.venv/bin/python -m pytest apps/server/tests/test_runtime_service.py -q`

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
