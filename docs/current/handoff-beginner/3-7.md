## 상세 인수인계

### 이 문서에서 이해해야 할 것

Redis 문서는 “서버를 여러 번 재시작해도 게임 흐름을 얼마나 복구할 수 있는가”를 이해하는 문서입니다. Redis는 게임 규칙을 판단하지 않고, 이미 검증된 세션/방/스트림/프롬프트/런타임 체크포인트를 저장합니다. 따라서 Redis 문제를 볼 때는 먼저 “저장 실패인가, 저장된 데이터를 해석하는 런타임 문제인가”를 분리해야 합니다.

### 관련 코드와 문서

### 실제 처리 흐름과 기대 값

- 이 문서의 흐름: Redis.
- 진입점: Redis는 세션/stream/replay 같은 서버 상태 저장과 배포 환경 공유 상태에 쓰입니다.
- 내부 처리: 규칙 판정은 Redis가 하지 않고 엔진/서비스가 한 결과를 저장하거나 재전송 가능한 형태로 남깁니다.
- 기대 값/검증 포인트: 기대 값은 연결 가능 여부, stream seq 보존, TTL/복구 정책입니다.

### 수정할 때 주의할 점

- Redis에 값이 있다는 사실과 그 값이 최신이라는 사실은 다릅니다. seq, checkpoint version, projection freshness를 같이 확인하세요.
- private 정보가 Redis canonical stream에 그대로 저장되면 안 됩니다. viewer별 projection은 배달/캐시 단계에서 만들어야 합니다.
- Redis 장애처럼 보여도 실제로는 런타임 lease, prompt replay, stale view_state 문제일 수 있습니다.

### 확인 방법

- `.venv/bin/python -m pytest apps/server/tests/test_redis_realtime_services.py -q`
- `.venv/bin/python -m pytest apps/server/tests/test_stream_service.py -q`

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
