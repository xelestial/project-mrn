## 상세 인수인계

### 이 문서에서 이해해야 할 것

초보자가 가장 먼저 헷갈리는 지점은 “화면에서 누른 버튼”과 “서버/엔진에서 실제로 처리되는 지점”이 떨어져 있다는 점입니다. 이 문서는 화면 행동을 기준으로 프론트, API/WebSocket, 백엔드 서비스, 엔진, 기대 값을 한 줄로 묶어 읽는 지도입니다.

### 공통으로 확인할 값

- 로비/시작 흐름: `room_no`, `room_token`, `seat`, `ready`, `session_id`, `session_token`
- 실시간 진행 흐름: `seq`, `request_id`, `player_id`, `module_id`, `active_module_id`, `view_state`
- prompt 흐름: `request_type`, `choices`, `resume_token`, `frame_id`, `module_type`, `decision_ack.status`
- 화면 검증: `LobbyView`, `PromptOverlay`, `BoardPanel`, `PlayersPanel`, `TimelinePanel`, `GameEventOverlay`가 같은 사건을 같은 값으로 표시하는지 확인합니다.

### 서버 연결/방 목록 새로고침

- 사용자가 누르는 것: 로비의 서버 연결 또는 방 목록 새로고침 버튼
- 프론트 시작점: `LobbyView.tsx` 버튼 콜백 -> `App.tsx`의 연결/새로고침 핸들러
- 요청: `sessionApi.ts`의 `listRooms`, 필요하면 base URL 정규화
- 백엔드: `routes/rooms.py`의 방 목록 조회 -> `RoomService`
- 기대 값: `rooms[]`에 `room_no`, `room_title`, `human_joined_count`, `human_ready_count`, `status`가 있어야 하고, 화면의 공개 방 목록 개수와 일치해야 합니다.

### 방 만들기

- 사용자가 누르는 것: `친구용 방 만들기` 또는 방 만들기 패널의 `방 만들기`
- 프론트 시작점: `LobbyView.tsx` -> `App.tsx` `handleCreateRoom`
- 요청: `sessionApi.ts` `createRoom`
- 백엔드: `routes/rooms.py` `POST /api/v1/rooms` -> `RoomService.create_room`
- 기대 값: 응답에 `room_no`, `room_token`, host `seat`, 좌석별 `seat_type`, `ready=false`, `status=waiting`이 있어야 합니다. 화면은 `activeRoom`과 `activeRoomSeat`를 갱신하고 현재 방 패널을 보여야 합니다.

### 방 입장

- 사용자가 누르는 것: 공개 방 목록의 `N번 좌석 참가`
- 프론트 시작점: `LobbyView.tsx` -> `App.tsx` `handleJoinRoom`
- 요청: `sessionApi.ts` `joinRoom(roomNo, seat, nickname)`
- 백엔드: `routes/rooms.py` `POST /api/v1/rooms/{room_no}/join` -> `RoomService.join_room`
- 기대 값: 응답에 새 `room_token`, `player_id`, `seat`가 있고 해당 좌석의 `player_id`가 채워져야 합니다. 내 좌석 표시와 준비 버튼 대상이 같은 seat여야 합니다.

### 준비/준비 해제

- 사용자가 누르는 것: 현재 방 패널의 준비 토글
- 프론트 시작점: `LobbyView.tsx` -> `App.tsx` `handleToggleReady`
- 요청: `sessionApi.ts` `setRoomReady(roomNo, roomToken, ready)`
- 백엔드: `routes/rooms.py` `POST /api/v1/rooms/{room_no}/ready` -> `RoomService.set_ready`
- 기대 값: 내 좌석의 `ready`가 요청값과 같아야 하고, `human_ready_count`가 화면의 준비 인원 표시와 일치해야 합니다. 호스트의 시작 버튼 활성 조건은 사람 좌석이 모두 입장하고 모두 준비된 상태입니다.

### 방 시작

- 사용자가 누르는 것: 호스트의 게임 시작 버튼
- 프론트 시작점: `LobbyView.tsx` -> `App.tsx` `handleStartRoom`
- 요청: `sessionApi.ts` `startRoom(roomNo, roomToken)`
- 백엔드: `routes/rooms.py` `POST /api/v1/rooms/{room_no}/start` -> `RoomService.start_room` -> `SessionService.start_session` -> `StreamService.publish(session_start/session_started)` -> `RuntimeService.start_runtime`
- 기대 값: 응답에 `session_id`와 좌석별 `session_tokens`가 있어야 합니다. 프론트는 내 좌석의 `session_token`을 저장하고 `#/match?session=...&token=...`로 이동해야 합니다.

### 게임 화면 WebSocket 연결

- 사용자가 하는 것: 매치 화면 진입, 새로고침, 또는 끊긴 뒤 재연결
- 프론트 시작점: `App.tsx`가 `sessionId/sessionToken`을 `useGameStream`에 전달
- 요청: `StreamClient.connect`가 `WS /api/v1/sessions/{session_id}/stream?token=...` 연결
- 백엔드: `routes/stream.py` `stream_ws` -> `SessionService.verify_session_token` -> `StreamService.subscribe`
- 기대 값: heartbeat가 내려오고 `seq`가 최신값으로 맞춰져야 합니다. heartbeat payload의 `runner_kind`, `active_module_id`, backpressure 값은 복구/디버깅 때 보는 힌트입니다.

### WebSocket resume/replay

- 사용자가 하는 것: 화면을 새로고침하거나 네트워크가 잠깐 끊겼다가 돌아옴
- 프론트 시작점: `useGameStream`의 gap/reconnect 처리
- 요청: WebSocket 메시지 `type=resume`, `last_seq`
- 백엔드: `routes/stream.py` resume 분기 -> `StreamService.replay_window` -> `StreamService.replay_from` -> viewer projection
- 기대 값: `last_seq` 이후 메시지가 순서대로 다시 내려와야 합니다. gap이 replay window보다 오래되면 `RESUME_GAP_TOO_OLD` error와 최신 버퍼 기준 복구가 발생할 수 있습니다.

### 프롬프트 선택

- 사용자가 누르는 것: `PromptOverlay`의 선택지, 카드, 대상, 확인/스킵 버튼
- 프론트 시작점: `selectActivePrompt` -> `PromptOverlay` -> `useGameStream.sendDecision`
- 요청: WebSocket 메시지 `type=decision`, `request_id`, `player_id`, 선택 payload
- 백엔드: `routes/stream.py` decision 분기 -> player 검증 -> `PromptService.submit_decision` -> `StreamService.publish(decision_ack)`
- 엔진: `DecisionGateway`가 같은 `request_id`의 decision을 소비하면 해당 runtime frame/module이 재개됩니다.
- 기대 값: `decision_ack.status=accepted`는 “접수됨”입니다. 실제 게임 변화는 뒤따르는 `decision_resolved`, event, `view_state`에서 확인해야 합니다.

### 캐릭터 드래프트/최종 선택

- 사용자가 누르는 것: 드래프트 후보 캐릭터 또는 최종 캐릭터 선택
- 프론트 시작점: `PromptOverlay`, `promptSurfaceCatalog.ts`, `promptSelectors.ts`
- 요청: WebSocket decision, 보통 `request_type=draft_card`, `final_character`, `final_character_choice` 계열
- 백엔드/엔진: `PromptService`/`DecisionGateway` -> 드래프트 관련 runtime module -> 캐릭터 상태 commit
- 기대 값: 본인에게는 선택 가능한 후보와 결과가 보여야 하고, 다른 좌석에는 숨겨야 할 후보/선택 정보가 visibility projection 뒤에 노출되면 안 됩니다.

### 턴 시작 마크/지목

- 사용자가 누르는 것: 지목 대상 캐릭터 선택 또는 스킵
- 프론트 시작점: `PromptOverlay` + `selectMarkTargetCharacterSlots`
- 요청: WebSocket decision, 대상은 플레이어 번호보다 캐릭터/슬롯 의미가 우선입니다.
- 백엔드/엔진: `PromptService`/`DecisionGateway` -> `turn_modules.py`, `sequence_modules.py`, `modifiers.py`
- 기대 값: `mark_queued`, `mark_resolved`, `mark_blocked`, `mark_target_none` 같은 이벤트 중 하나로 결과가 설명되어야 합니다. 화면의 대상 표시와 stream event의 대상이 같아야 합니다.

### 잔꾀 사용

- 사용자가 누르는 것: 잔꾀 카드 사용/스킵/카드 선택
- 프론트 시작점: `PromptOverlay`, `PlayerTrickPeek`, `promptSelectors.ts`
- 요청: WebSocket decision
- 백엔드/엔진: `PromptService`/`DecisionGateway` -> `GPT/runtime_modules/sequence_modules.py` -> `GPT/trick_cards.py`
- 기대 값: `trick_used` 또는 skip 결과, 손패 변화, 후속 이동/도착/prompt가 이어져야 합니다. 잔꾀는 한 번의 카드 사용으로 끝나지 않고 frame stack에 후속 처리를 만들 수 있습니다.

### 주사위와 이동

- 사용자가 누르는 것: 주사위 카드/이동 선택이 있는 경우의 prompt 선택. 자동 이동이면 사용자가 누르는 버튼이 없습니다.
- 프론트 시작점: `PromptOverlay` 또는 stream event를 받은 `BoardPanel`
- 요청: 선택이 있으면 WebSocket decision, 자동이면 런타임 진행 이벤트
- 백엔드/엔진: `sequence_modules.py`, 주사위/이동 모듈, `tile_effects.py`
- 기대 값: `dice_roll`의 눈/합계, `player_move`의 출발/도착 칸, 보드 projection의 말 위치가 일치해야 합니다. 프론트 애니메이션은 서버 경로를 표시할 뿐 이동량을 새로 계산하지 않습니다.

### 도착 처리/구매/통행료/운수

- 사용자가 누르는 것: 구매 여부, 보상 선택, 운수 관련 선택 prompt
- 프론트 시작점: `PromptOverlay`, `SituationPanel`, `PlayersPanel`, `GameEventOverlay`
- 요청: 필요한 경우 WebSocket decision
- 백엔드/엔진: `tile_effects.py`, `fortune_cards.py`, 경제/소유권 처리 모듈
- 기대 값: `tile_purchased`, `rent_paid`, `fortune_drawn`, `fortune_resolved`, 현금/조각/승점 변화가 같은 seq 흐름 안에서 설명되어야 합니다.

### 보급/동시 선택

- 사용자가 누르는 것: 보급 선택 prompt의 선택지
- 프론트 시작점: 각 좌석의 `PromptOverlay`
- 요청: 플레이어별 WebSocket decision
- 백엔드/엔진: `GPT/runtime_modules/simultaneous.py`, `prompts.py`, `PromptService`
- 기대 값: 모든 필수 플레이어의 decision이 모이기 전에는 결과가 확정되면 안 됩니다. 완료 뒤에는 보급 결과 이벤트와 손패/자원 변화가 한 번에 맞아야 합니다.

### 라운드 종료/카드 플립

- 사용자가 하는 것: 별도 버튼이 아니라 마지막 턴/보급 이후 stream을 관찰합니다.
- 프론트 시작점: `TimelinePanel`, `CoreActionPanel`, `GameEventOverlay`
- 요청: 사용자 요청 없음. 런타임 자동 진행입니다.
- 백엔드/엔진: `round_modules.py`의 라운드 종료/카드 플립 모듈 -> `StreamService.publish`
- 기대 값: 라운드 종료 이벤트, 공개 카드 변화, 다음 라운드 시작 이벤트가 `seq` 순서대로 내려와야 합니다.

### 게임 종료/아카이브

- 사용자가 하는 것: 종료 조건 도달 뒤 최종 결과/로그를 확인합니다.
- 프론트 시작점: stream snapshot을 받은 결과/상태 컴포넌트, replay/log 화면
- 요청: 필요하면 replay export 또는 runtime status 조회
- 백엔드/엔진: runtime 종료 상태 -> `StreamService` -> `archive_service.py`/replay 관련 서비스
- 기대 값: 최종 점수/승자/종료 사유, 마지막 `seq`, 재현 가능한 replay가 남아야 합니다. 종료 뒤 새 decision이 accepted 되면 안 됩니다.

### 버그를 볼 때의 빠른 판정법

- 화면 버튼이 문제면 먼저 `LobbyView.tsx` 또는 `PromptOverlay`에서 시작합니다.
- HTTP 요청이 문제면 `sessionApi.ts` 함수명과 `routes/rooms.py`/`routes/sessions.py` endpoint를 맞춥니다.
- WebSocket decision이 문제면 `request_id`, `player_id`, `decision_ack.status`를 봅니다.
- 엔진 진행이 문제면 heartbeat의 `active_module_id`, runtime checkpoint의 `runtime_frame_stack`, stream event의 `module_id`를 봅니다.
- 화면 표시가 문제면 백엔드 `view_state`와 프론트 selector 결과가 같은 의미인지 비교합니다.
