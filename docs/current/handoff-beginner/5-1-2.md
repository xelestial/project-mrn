다른 플레이어는 `POST /api/v1/rooms/{room_no}/join`으로 좌석과 닉네임을 보냅니다. 성공하면 자신의 `room_member_token`을 받습니다.

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 API 흐름 중 `방 입장`을 설명합니다. API를 읽을 때는 요청/응답 필드만 보지 말고, 그 호출 다음에 어떤 stream event와 화면 변화가 따라오는지 같이 봐야 합니다.

### 관련 코드와 문서

- `docs/current/api/online-game-api-spec.md`
- `apps/server/src/routes/`
- `apps/web/src/infra/http/sessionApi.ts`
- `apps/web/src/infra/ws/StreamClient.ts`
- `apps/web/src/hooks/useGameStream.ts`

### 실제 처리 흐름과 기대 값

- 이 문서의 API 흐름: 방 입장.
- 사용자 행동/요청: 공개 방 목록에서 빈 사람 좌석을 누르면 `joinRoom(roomNo, seat)`이 나갑니다.
- 서버 처리: `POST /api/v1/rooms/{room_no}/join` -> `RoomService.join_room`이 좌석 점유와 room token을 처리합니다.
- 기대 값/검증 포인트: 선택한 seat의 `player_id`가 채워지고, 클라이언트가 받은 `room_token`이 이후 ready/start/leave 요청에 사용되어야 합니다.

### 수정할 때 주의할 점

- REST는 시작/조회/복구에, WebSocket은 실시간 게임 진행에 더 가깝습니다.
- 모든 에러는 ok/data/error envelope 기준으로 읽으세요.
- API 응답과 이후 stream event를 한 흐름으로 기록하면 디버깅이 쉬워집니다.

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
