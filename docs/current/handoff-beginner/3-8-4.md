서버는 주기적으로 heartbeat를 보냅니다. heartbeat에는 최신 seq, backpressure, runner kind, active module id 같은 진단 정보가 들어갑니다.

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 백엔드 관점의 `heartbeat`을 설명합니다. 백엔드는 API 요청, 엔진 실행, stream 발행, 저장소 연결, visibility projection 사이의 접착층입니다.

### 관련 코드와 문서

- `apps/server/src/app.py`
- `apps/server/src/state.py`
- `apps/server/src/routes/`
- `apps/server/src/services/`
- `apps/server/src/domain/view_state/`
- `apps/server/src/routes/stream.py`
- `apps/server/src/services/stream_service.py`
- `apps/web/src/infra/ws/StreamClient.ts`

### 실제 처리 흐름과 기대 값

- 이 문서의 흐름: heartbeat.
- 진입점: WebSocket 연결이 살아 있는 동안 서버 heartbeat task가 주기적으로 실행됩니다.
- 내부 처리: 최신 seq, backpressure, runtime status, active module id를 읽어 payload로 보냅니다.
- 기대 값/검증 포인트: 화면은 연결 유지/복구 판단에 heartbeat를 쓰고, 디버깅 때 `active_module_id`로 멈춘 모듈을 찾을 수 있어야 합니다.

### 수정할 때 주의할 점

- WebSocket 메시지는 반드시 seq 순서로 읽으세요.
- 연결이 끊긴 뒤에는 마지막 seq로 resume하고, gap이 너무 오래되면 replay export로 최신 view_state를 받습니다.
- 관전자와 플레이어는 같은 stream을 보더라도 visibility projection 결과가 다를 수 있습니다.

### 확인 방법

- `npm --prefix apps/web run test -- src/hooks/useGameStream.spec.ts src/infra/ws/StreamClient.spec.ts --run`

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
