`apps/server/src/services/realtime_persistence.py`의 `RedisStreamStore`는 Redis `XADD`로 이벤트를 저장하고, `seq`는 별도 key에서 증가시킵니다. WebSocket replay는 이 stream history에서 `last_seq` 이후 메시지를 다시 보냅니다.

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 백엔드 관점의 `Redis stream`을 설명합니다. 백엔드는 API 요청, 엔진 실행, stream 발행, 저장소 연결, visibility projection 사이의 접착층입니다.

### 관련 코드와 문서

- `apps/server/src/app.py`
- `apps/server/src/state.py`
- `apps/server/src/routes/`
- `apps/server/src/services/`
- `apps/server/src/domain/view_state/`
- `apps/server/src/infra/redis_client.py`
- `apps/server/src/services/realtime_persistence.py`
- `apps/server/src/services/persistence.py`

### 실제 처리 흐름과 기대 값

- 이 문서의 흐름: Redis stream.
- 진입점: `StreamService.publish`가 새 메시지에 seq를 붙여 저장/전송할 때입니다.
- 내부 처리: Redis stream 또는 호환 저장소가 replay window를 유지하고 WebSocket 구독자에게 전달할 메시지를 보관합니다.
- 기대 값/검증 포인트: `seq`가 단조 증가하고, `replay_from(last_seq)`가 빠진 메시지를 순서대로 돌려줘야 합니다.

### 수정할 때 주의할 점

- Redis에 값이 있다는 사실과 그 값이 최신이라는 사실은 다릅니다. seq, checkpoint version, projection freshness를 같이 확인하세요.
- private 정보가 Redis canonical stream에 그대로 저장되면 안 됩니다. viewer별 projection은 배달/캐시 단계에서 만들어야 합니다.
- Redis 장애처럼 보여도 실제로는 런타임 lease, prompt replay, stale view_state 문제일 수 있습니다.

### 확인 방법

- `.venv/bin/python -m pytest apps/server/tests/test_stream_service.py -q`

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
