프론트는 `decision` 메시지에 `request_id`, `player_id`, `choice_id`, `choice_payload`, continuation 정보를 담아 보냅니다.

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 API 흐름 중 `프론트가 decision 전송`을 설명합니다. API를 읽을 때는 요청/응답 필드만 보지 말고, 그 호출 다음에 어떤 stream event와 화면 변화가 따라오는지 같이 봐야 합니다.

### 관련 코드와 문서

- `docs/current/api/online-game-api-spec.md`
- `apps/server/src/routes/`
- `apps/web/src/infra/http/sessionApi.ts`
- `apps/web/src/infra/ws/StreamClient.ts`
- `apps/web/src/hooks/useGameStream.ts`
- `apps/server/src/services/prompt_service.py`
- `apps/server/src/services/decision_gateway.py`

### 실제 처리 흐름과 기대 값

- 이 문서의 API 흐름: 프론트가 decision 전송.
- 사용자 행동/요청: `PromptOverlay`에서 선택지를 누르면 `useGameStream.sendDecision`이 WebSocket 메시지를 보냅니다.
- 서버 처리: `routes/stream.py`가 seat/player를 검증하고 `PromptService.submit_decision`에 저장합니다.
- 기대 값/검증 포인트: 메시지의 `request_id`와 prompt의 `request_id`가 같고, `player_id`가 인증된 seat와 같아야 accepted 됩니다.

### 수정할 때 주의할 점

- prompt의 request_id와 decision의 request_id가 같은지 먼저 확인하세요.
- 모듈형 런타임 prompt는 resume_token, frame_id, module_id, module_type이 중요합니다.
- decision_ack는 “접수 여부”이고, 실제 게임 상태 변화는 이후 event/view_state로 확인해야 합니다.

### 확인 방법

- `.venv/bin/python -m pytest apps/server/tests/test_sessions_api.py apps/server/tests/test_stream_api.py -q`
- `npm --prefix apps/web run test -- src/infra/http --run`

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
