# 3-6. Visibility projection

`apps/server/src/domain/visibility/`는 관전자와 특정 플레이어가 볼 수 있는 정보를 분리합니다. 비공개 손패나 특정 플레이어용 프롬프트는 대상 플레이어에게만 보여야 하므로, 프론트에서 숨기는 방식이 아니라 백엔드 projection에서 먼저 걸러야 합니다.

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 백엔드 관점의 `Visibility projection`을 설명합니다. 백엔드는 API 요청, 엔진 실행, stream 발행, 저장소 연결, visibility projection 사이의 접착층입니다.

### 관련 코드와 문서

- `apps/server/src/app.py`
- `apps/server/src/state.py`
- `apps/server/src/routes/`
- `apps/server/src/services/`
- `apps/server/src/domain/view_state/`

### 실제 처리 흐름과 기대 값

- 이 문서의 흐름: visibility projection은 같은 stream event를 보는 사람에 따라 다르게 보이게 만드는 경계입니다.
- 진입점: `StreamService.project_message_for_viewer`와 stream route의 replay/project 경로입니다.
- 내부 처리: 좌석 토큰, spectator 여부, player id를 기준으로 비공개 손패/선택지/view_state를 제거하거나 축약합니다.
- 기대 값/검증 포인트: 본인에게 필요한 private 정보는 보여야 하지만, 다른 좌석이나 관전자에게 숨겨야 할 값은 replay에서도 노출되면 안 됩니다.

### 수정할 때 주의할 점

- 라우터는 얇게, 실제 상태 변경은 서비스에서 찾으세요.
- stream payload에는 semantic guard와 visibility projection 경계가 걸려야 합니다.
- 복구 문제는 runtime-status, replay, Redis store를 함께 확인하세요.

### 확인 방법

- `python3 tools/plan_policy_gate.py`
- 관련 서버/프론트/엔진 테스트를 함께 실행하세요.

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
