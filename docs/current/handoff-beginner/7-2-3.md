`apps/server/src/infra/structured_log.py`를 통해 세션 생성, 연결, decision 수신, runtime 시작 같은 서버 이벤트가 기록됩니다.

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 로비/로그 관점의 `Backend structured log`을 설명합니다. 로비는 게임 시작 전 좌석과 준비 상태를 관리하고, 로그는 게임 중 발생한 일을 다시 설명하는 증거입니다.

### 관련 코드와 문서

- `apps/server/src/routes/rooms.py`
- `apps/server/src/services/room_service.py`
- `apps/server/src/routes/debug.py`
- `apps/server/src/infra/game_debug_log.py`
- `apps/web/src/features/lobby/LobbyView.tsx`
- `apps/web/src/App.tsx`

### 실제 처리 흐름과 기대 값

- 이 문서의 로그 흐름: 화면에서 이상한 장면을 본 순간의 `seq`를 먼저 잡고, prompt 문제면 `request_id`, 런타임 문제면 `module_id`로 좁힙니다.
- 연결 경로: stream message는 WebSocket/replay의 원본 기록이고, frontend debug log는 화면 해석 기록이며, backend structured log/game debug log는 서버 처리 기록입니다.
- 판정 방식: 같은 사건이 `seq -> request_id -> module_id -> view_state` 순서로 이어지는지 확인합니다.
- 기대 값/검증 포인트: 로그마다 이름은 달라도 같은 세션, 같은 seq 범위, 같은 prompt/event를 가리켜야 합니다.

### 수정할 때 주의할 점

- 방(room)과 세션(session)을 섞어 말하면 디버깅이 어려워집니다.
- 로그는 seq, request_id, module_id, view_state 순서로 좁히세요.
- 프론트 debug log는 화면에서 일어난 일을 서버 로그와 맞추기 위한 보조 자료입니다.

### 확인 방법

- `.venv/bin/python -m pytest apps/server/tests -k "room or replay or stream" -q`
- `npm --prefix apps/web run test -- src/infra/http src/hooks --run`

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
