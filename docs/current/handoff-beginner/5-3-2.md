서버는 연결을 등록하고, heartbeat를 보내며, 새 event/prompt가 publish될 때마다 viewer에 맞게 projection한 메시지를 보냅니다.

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 API 흐름 중 `최초 연결 직후`을 설명합니다. API를 읽을 때는 요청/응답 필드만 보지 말고, 그 호출 다음에 어떤 stream event와 화면 변화가 따라오는지 같이 봐야 합니다.

### 관련 코드와 문서

- `docs/current/api/online-game-api-spec.md`
- `apps/server/src/routes/`
- `apps/web/src/infra/http/sessionApi.ts`
- `apps/web/src/infra/ws/StreamClient.ts`
- `apps/web/src/hooks/useGameStream.ts`
- `apps/server/src/routes/stream.py`
- `apps/server/src/services/stream_service.py`

### 실제 처리 흐름과 기대 값

- 이 문서의 API 흐름: 최초 연결 직후.
- 사용자 행동/요청: 연결이 열리면 클라이언트는 수신 상태를 초기화하고 최신 stream을 기다립니다.
- 서버 처리: `StreamService.subscribe` 등록 뒤 heartbeat task와 sender task가 동시에 돌며 viewer별 projected message를 보냅니다.
- 기대 값/검증 포인트: 첫 heartbeat의 `seq`, `runner_kind`, `active_module_id`가 화면 연결 상태와 디버그 정보에 반영되어야 합니다.

### 수정할 때 주의할 점

- WebSocket 메시지는 반드시 seq 순서로 읽으세요.
- 연결이 끊긴 뒤에는 마지막 seq로 resume하고, gap이 너무 오래되면 replay export로 최신 view_state를 받습니다.
- 관전자와 플레이어는 같은 stream을 보더라도 visibility projection 결과가 다를 수 있습니다.

### 확인 방법

- `npm --prefix apps/web run test -- src/hooks/useGameStream.spec.ts src/infra/ws/StreamClient.spec.ts --run`

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
