최종 선택 단계에서 선택지가 사실상 1개만 남으면 별도 프롬프트 없이 자동 처리될 수 있습니다. 사용자는 "왜 선택창이 안 떴지?"라고 느낄 수 있는데, 이 경우는 엔진이 자동 확정한 정상 흐름일 수 있습니다.

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 게임 규칙 중 `최종 캐릭터 선택`을 설명합니다. 규칙 문서는 UI 문구가 아니라 엔진이 실제로 따라야 하는 계약으로 읽어야 합니다. 화면에서 그럴듯하게 보여도 엔진 이벤트와 테스트가 다르면 규칙 구현이 맞다고 볼 수 없습니다.

### 관련 코드와 문서

- `docs/current/Game-Rules.md`
- `GPT/runtime_modules/round_modules.py`
- `GPT/runtime_modules/turn_modules.py`
- `GPT/engine.py` (모듈에서 아직 호출하는 레거시 규칙 helper 확인용)
- `GPT/test_rule_fixes.py`

### 실제 처리 흐름과 기대 값

- 이 문서의 규칙 흐름: 최종 캐릭터 선택.
- 게임 안에서 시작되는 지점: 드래프트 후보가 모인 뒤 플레이어가 이번 게임에서 사용할 캐릭터를 확정할 때 시작됩니다.
- 처리 방식: 최종 선택 prompt -> WebSocket decision -> 엔진 commit -> 캐릭터 슬롯/능력 상태 stream 발행 순서입니다.
- 기대 값/화면 결과: 최종 `character_id`와 플레이어 매핑이 확정되고, 다른 좌석에는 숨겨야 할 선택 과정이 노출되지 않아야 합니다.

### 수정할 때 주의할 점

- 규칙을 바꾸면 프론트 표시보다 엔진 테스트를 먼저 고치거나 추가하세요.
- 카드 이름 문자열보다 card id, trait, module metadata 같은 식별자를 우선 사용하세요.
- 즉시 효과인지, 대상 턴 시작에 해결되는 효과인지 구분하세요.

### 확인 방법

- `.venv/bin/python -m pytest GPT/test_rule_fixes.py -q`
- `.venv/bin/python -m pytest apps/server/tests/test_runtime_service.py -q`

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
