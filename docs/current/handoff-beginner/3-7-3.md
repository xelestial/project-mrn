## 상세 인수인계

### 이 문서에서 이해해야 할 것

저장 대상은 크게 네 묶음입니다. 첫째, 방과 세션처럼 사용자가 다시 들어올 때 필요한 식별 정보입니다. 둘째, stream event와 seq처럼 화면 재구성에 필요한 순서 정보입니다. 셋째, pending/resolved prompt와 accepted decision command처럼 사람 선택을 잃지 않기 위한 정보입니다. 넷째, runtime/game checkpoint와 projected view_state처럼 재시작 후 현재 장면을 복원하기 위한 정보입니다.

### 실제 처리 흐름과 기대 값

- 이 문서의 흐름: Redis에 저장되는 것.
- 진입점: 세션 상태 변경, stream publish, prompt decision 접수, replay 저장 순간입니다.
- 내부 처리: 세션/room/runtime 보조 상태와 stream item이 key 또는 stream 구조로 저장됩니다.
- 기대 값/검증 포인트: `session_id`와 `seq` 기준으로 최신 상태를 찾을 수 있어야 하고, private payload 저장/노출 경계를 확인해야 합니다.

### 수정할 때 주의할 점

- Redis에 값이 있다는 사실과 그 값이 최신이라는 사실은 다릅니다. seq, checkpoint version, projection freshness를 같이 확인하세요.
- private 정보가 Redis canonical stream에 그대로 저장되면 안 됩니다. viewer별 projection은 배달/캐시 단계에서 만들어야 합니다.
- Redis 장애처럼 보여도 실제로는 런타임 lease, prompt replay, stale view_state 문제일 수 있습니다.

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
