## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 엔진/런타임 관점의 `엔진`을 설명합니다. 현재 엔진은 런타임 진행 구조가 모듈화되어 있으며, `engine.py`는 새 구조의 중심 파일이라기보다 `ModuleRunner`가 아직 위임하는 기존 규칙 helper와 semantic kernel에 가깝습니다. 그래서 새 개발자는 먼저 `runtime_modules`의 세부 파일을 읽고, 필요한 경우에만 `engine.py`로 내려가야 합니다.

### 관련 코드와 문서

- `GPT/runtime_modules/contracts.py`
- `GPT/runtime_modules/runner.py`
- `GPT/runtime_modules/round_modules.py`
- `GPT/runtime_modules/turn_modules.py`
- `GPT/runtime_modules/sequence_modules.py`
- `GPT/runtime_modules/simultaneous.py`
- `GPT/engine.py` (모듈 런타임이 아직 위임하는 기존 규칙 helper)
- `apps/server/src/services/runtime_service.py`
- `docs/current/runtime/end-to-end-contract.md`

### 실제 처리 흐름과 기대 값

- 이 문서의 엔진 흐름: 2. 엔진.
- 시작 지점: 백엔드의 `RuntimeService.start_runtime`이 엔진과 모듈 runner를 만들고, prompt가 필요한 순간에는 런타임이 멈춥니다.
- 결정 재개: 사용자의 WebSocket decision은 `PromptService`에 저장되고 `DecisionGateway`가 같은 `request_id`를 소비한 뒤 해당 frame/module을 재개합니다.
- 기대 값/검증 포인트: checkpoint에는 `runtime_frame_stack`, `active_module_id`, prompt continuation 정보가 남아야 하고, 결과는 event와 `view_state`로만 프론트에 전달되어야 합니다.

### 수정할 때 주의할 점

- `engine.py`에 새 단계를 바로 추가하기보다, 먼저 그 단계가 round/turn/sequence/simultaneous 중 어느 모듈에 속하는지 정하세요.
- 런타임 단계가 꼬이면 frame/module/request_id를 같이 추적하세요.
- 프롬프트가 뜨는 단계에서는 엔진이 잠시 멈춘 것이 정상입니다.
- 자동 선택과 사람 선택이 같은 상태 전이 결과를 만들어야 합니다.

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
