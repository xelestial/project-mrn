`useGameStream()`은 WebSocket gap을 감지하면 `resume`을 요청하고, decision 전송 이후에는 여러 지연 타이머로 replay fetch를 시도해 UI를 최신 상태로 당깁니다.

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 API 흐름 중 `프론트 recovery`을 설명합니다. API를 읽을 때는 요청/응답 필드만 보지 말고, 그 호출 다음에 어떤 stream event와 화면 변화가 따라오는지 같이 봐야 합니다.

### 관련 코드와 문서

- `docs/current/api/online-game-api-spec.md`
- `apps/server/src/routes/`
- `apps/web/src/infra/http/sessionApi.ts`
- `apps/web/src/infra/ws/StreamClient.ts`
- `apps/web/src/hooks/useGameStream.ts`

### 실제 처리 흐름과 기대 값

- 이 문서의 API 흐름: 프론트 recovery.
- 사용자 행동/요청: 프론트가 gap, reconnect, 새로고침을 감지하면 저장된 token과 last seq로 복구를 시도합니다.
- 서버 처리: WebSocket resume, replay, runtime status 조회를 조합해 최신 snapshot을 되찾습니다.
- 기대 값/검증 포인트: 중복 event는 무시되고 누락 event는 채워지며, 화면은 같은 `session_id`의 최신 `view_state`로 돌아와야 합니다.

### 수정할 때 주의할 점

- REST는 시작/조회/복구에, WebSocket은 실시간 게임 진행에 더 가깝습니다.
- 모든 에러는 ok/data/error envelope 기준으로 읽으세요.
- API 응답과 이후 stream event를 한 흐름으로 기록하면 디버깅이 쉬워집니다.

- `npm --prefix apps/web run test -- src/infra/http --run`

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
