## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 `규칙 버그` 항목을 독립적으로 설명합니다.

### 관련 코드와 문서

- `docs/current/Game-Rules.md`
- `docs/current/api/online-game-api-spec.md`
- `apps/server/tests/`
- `apps/web/src/`
- `GPT/test_*.py`

### 실제 처리 흐름과 기대 값

- 이 문서의 작업 흐름: 버그를 규칙/API/UI/Redis/WebSocket 중 어디에서 시작됐는지 먼저 한 갈래로 고릅니다.
- 추적 경로: 화면 행동과 `seq/request_id/module_id`를 연결한 뒤 해당 계층의 테스트와 로그를 같이 봅니다.
- 수정 방식: 원인 계층 하나를 고쳐도 프론트 표시, stream event, replay, runtime checkpoint가 같은 결과를 말하는지 확인합니다.
- 기대 값/검증 포인트: 재현 절차, 수정 파일, 기대 event/view_state, 회귀 테스트가 문서나 PR 설명에 남아야 합니다.

### 수정할 때 주의할 점

- 상위 문서와 하위 문서를 함께 읽고, 관련 테스트를 먼저 찾으세요.

### 확인 방법

- `python3 tools/plan_policy_gate.py`
- 관련 서버/프론트/엔진 테스트를 함께 실행하세요.

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
