# 3-5. 백엔드 selector와 view_state

`apps/server/src/domain/view_state/`는 stream history를 프론트가 쓰기 쉬운 `view_state`로 바꾸는 계층입니다. 보드, 플레이어, 손패, 프롬프트, 턴 상태, 장면 상태 등이 여기서 만들어집니다.

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 백엔드 관점의 `백엔드 selector와 view_state`을 설명합니다. 백엔드는 API 요청, 엔진 실행, stream 발행, 저장소 연결, visibility projection 사이의 접착층입니다.

### 관련 코드와 문서

- `apps/server/src/app.py`
- `apps/server/src/state.py`
- `apps/server/src/routes/`
- `apps/server/src/services/`
- `apps/server/src/domain/view_state/`

### 실제 처리 흐름과 기대 값

- 이 문서의 흐름: 백엔드 view_state selector는 엔진 상태를 화면용 스냅샷으로 투영합니다.
- 진입점: 런타임이 event를 만들거나 stream publish를 준비할 때 selector가 호출됩니다.
- 내부 처리: 원본 상태에서 viewer에게 필요한 공개 필드만 고르고, visibility projection 전후의 의미가 깨지지 않게 정리합니다.
- 기대 값/검증 포인트: 프론트 selector가 다시 규칙을 추론하지 않아도 말 위치, 자원, 현재 prompt, 공개 이벤트를 표시할 수 있어야 합니다.

### 수정할 때 주의할 점

- 라우터는 얇게, 실제 상태 변경은 서비스에서 찾으세요.
- stream payload에는 semantic guard와 visibility projection 경계가 걸려야 합니다.
- 복구 문제는 runtime-status, replay, Redis store를 함께 확인하세요.

### 확인 방법

- `python3 tools/plan_policy_gate.py`
- 관련 서버/프론트/엔진 테스트를 함께 실행하세요.

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
