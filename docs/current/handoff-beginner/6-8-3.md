필요한 선택은 prompt로 보내고, 결과는 event와 `view_state`로 보냅니다.

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 `도착 처리` 기능을 따라갑니다. 도착 칸은 구매/통행료/운수/특수 효과로 분기하고, 필요한 선택은 다시 prompt가 됩니다. 기능을 고칠 때는 한 계층만 보지 말고 엔진, 백엔드, 프론트 표시가 같은 흐름을 가리키는지 함께 확인해야 합니다.

### 관련 코드와 문서

- `GPT/runtime_modules/sequence_modules.py`
- `GPT/tile_effects.py`
- `GPT/fortune_cards.py`
- `apps/server/src/domain/view_state/`

### 실제 처리 흐름과 기대 값

- 이 문서의 흐름: 도착 처리.
- 사용자 행동/트리거: 말 이동이 끝나 도착 칸의 소유권/통행료/즉시 효과를 판정하는 흐름입니다.
- 프론트 경로: 프론트에서 새 행동을 누르는 흐름이라기보다 이동 이벤트 다음에 서버가 보낸 prompt나 event를 이어 읽습니다. 구매/보상 선택이 필요하면 `PromptOverlay` decision으로 돌아갑니다.
- 백엔드/엔진 경로: 엔진은 `tile_effects.py`, `fortune_cards.py`, 경제/소유권 처리 모듈에서 빈 땅, 자기 땅, 남의 땅, 즉시 효과 칸을 분기합니다.
- 기대 값/화면 결과: 기대 값은 `tile_purchased`, `rent_paid`, `fortune_drawn`/`fortune_resolved`, 현금/조각/승점 변화입니다. `SituationPanel`, `PlayersPanel`, `GameEventOverlay`가 같은 금액과 대상 플레이어를 보여야 합니다.

### 수정할 때 주의할 점

- 상위 문서와 하위 문서를 함께 읽고, 관련 테스트를 먼저 찾으세요.

### 확인 방법

- `python3 tools/plan_policy_gate.py`
- 관련 서버/프론트/엔진 테스트를 함께 실행하세요.

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
