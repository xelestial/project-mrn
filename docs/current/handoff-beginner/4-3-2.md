게임 화면은 `App.tsx`가 조립하고, 보드는 `features/board/BoardPanel.tsx`, 프롬프트는 `features/prompt/PromptOverlay.tsx`, 턴/상황/타임라인은 각각 `features/stage/`, `features/status/`, `features/timeline/` 아래 컴포넌트가 담당합니다.

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 프론트엔드 관점의 `게임 화면`을 설명합니다. 프론트는 규칙을 판단하는 곳이 아니라 서버가 준 prompt, event, view_state를 사람이 이해할 수 있게 보여주는 계층입니다.

### 관련 코드와 문서

- `apps/web/src/App.tsx`
- `apps/web/src/infra/http/sessionApi.ts`
- `apps/web/src/infra/ws/StreamClient.ts`
- `apps/web/src/hooks/useGameStream.ts`
- `apps/web/src/domain/selectors/`

### 실제 처리 흐름과 기대 값

- 이 문서의 흐름: 게임 화면.
- 화면에서 시작되는 지점: `App.tsx`가 stream snapshot을 selector에 넣고 보드/플레이어/프롬프트/타임라인 컴포넌트에 나눠 전달합니다.
- 상태 전달 방식: 컴포넌트는 규칙을 재계산하지 않고 selector 결과와 `view_state`를 표시합니다.
- 기대 값/화면 결과: 기대 값은 동일한 `seq` 기준의 말 위치, 현재 actor, prompt, 공개 이벤트입니다.

### 수정할 때 주의할 점

- 프론트에서 새 규칙을 계산하지 말고 prompt/view_state를 믿으세요.
- 화면만 고칠 때도 selector 테스트로 데이터 모델을 먼저 고정하세요.
- sessionId/token/baseUrl이 바뀌면 stream 상태와 중복 decision ledger를 초기화해야 합니다.

### 확인 방법

- `npm --prefix apps/web run test -- --run`
- `npm --prefix apps/web run build`

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
