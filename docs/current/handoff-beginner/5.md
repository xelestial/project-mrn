## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 API 흐름 중 `흐름으로 읽는 API 규약`을 설명합니다. API를 읽을 때는 요청/응답 필드만 보지 말고, 그 호출 다음에 어떤 stream event와 화면 변화가 따라오는지 같이 봐야 합니다.
특히 처음 읽는 사람은 [5-6. 화면 행동별 트리거 지도](5-6.md)를 먼저 열어 “내가 누른 것”이 어떤 프론트 함수, API/WS 메시지, 백엔드 서비스, 엔진 모듈, 기대 값을 거치는지 한 줄로 잡고 들어가면 됩니다.

### 관련 코드와 문서

- `docs/current/api/online-game-api-spec.md`
- `apps/server/src/routes/`
- `apps/web/src/infra/http/sessionApi.ts`
- `apps/web/src/infra/ws/StreamClient.ts`
- `apps/web/src/hooks/useGameStream.ts`

### 실제 처리 흐름과 기대 값

- 이 문서의 읽기 흐름: 5. 흐름으로 읽는 API 규약.
- 먼저 연결할 실제 흐름: 로비 행동은 `LobbyView.tsx -> sessionApi.ts -> routes/rooms.py`, 게임 중 선택은 `PromptOverlay -> useGameStream.sendDecision -> routes/stream.py -> PromptService/RuntimeService`로 이어집니다.
- 확인할 값: 시작/로비는 `room_no`, `room_token`, `ready`, `session_id`를 보고, 게임 진행은 `seq`, `request_id`, `module_id`, `view_state`를 봅니다.
- 기대 결과: 초보자는 추상 설명만 읽고 멈추지 말고, 각 기능 문서에서 버튼/요청/서비스/엔진/화면 값을 한 줄로 이어서 확인할 수 있어야 합니다.

### 수정할 때 주의할 점

- REST는 시작/조회/복구에, WebSocket은 실시간 게임 진행에 더 가깝습니다.
- 모든 에러는 ok/data/error envelope 기준으로 읽으세요.
- API 응답과 이후 stream event를 한 흐름으로 기록하면 디버깅이 쉬워집니다.

### 확인 방법

- `.venv/bin/python -m pytest apps/server/tests/test_sessions_api.py apps/server/tests/test_stream_api.py -q`
- `npm --prefix apps/web run test -- src/infra/http --run`

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
