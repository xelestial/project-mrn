SimultaneousResolutionFrame이 여러 플레이어의 동시 선택을 `batch_id`로 묶습니다.

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 `보급과 동시 선택` 기능을 따라갑니다. 여러 플레이어 선택은 batch_id로 묶이며 각 플레이어에게 자기 prompt만 보여야 합니다. 기능을 고칠 때는 한 계층만 보지 말고 엔진, 백엔드, 프론트 표시가 같은 흐름을 가리키는지 함께 확인해야 합니다.

### 관련 코드와 문서

- `GPT/runtime_modules/simultaneous.py`
- `GPT/runtime_modules/prompts.py`
- `apps/server/src/services/prompt_service.py`
- `apps/server/src/services/stream_service.py`

### 실제 처리 흐름과 기대 값

- 이 문서의 흐름: 보급과 동시 선택.
- 사용자 행동/트리거: 라운드 말이나 보급 조건에서 여러 플레이어가 동시에 선택해야 할 때 발생합니다.
- 프론트 경로: 각 플레이어 화면에는 자기 prompt만 보이고, 각자 WebSocket decision을 보냅니다. 관전자/다른 좌석에는 비공개 선택지가 노출되면 안 됩니다.
- 백엔드/엔진 경로: 엔진은 `GPT/runtime_modules/simultaneous.py`와 prompt 빌더에서 동시 선택 프레임을 만들고, 모든 필수 decision이 모이면 한 번에 결과를 commit합니다.
- 기대 값/화면 결과: 기대 값은 플레이어별 `request_id`, 접수된 선택 수, 최종 보급 결과 이벤트, 손패/자원 변화입니다. 일부만 응답한 상태에서는 결과가 먼저 적용되면 안 됩니다.

### 수정할 때 주의할 점

- 상위 문서와 하위 문서를 함께 읽고, 관련 테스트를 먼저 찾으세요.

### 확인 방법

- `python3 tools/plan_policy_gate.py`
- 관련 서버/프론트/엔진 테스트를 함께 실행하세요.

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
