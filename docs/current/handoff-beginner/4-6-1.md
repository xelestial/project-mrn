# 4-6-1. 규칙을 프론트에 새로 만들지 않는다

Status: ACTIVE
Updated: 2026-05-03

상위 목차: [인수인계 문서 지도](README.md)
문서 위치: [4. 프론트엔드](4.md) > [4-6. 프론트엔드를 고칠 때 주의할 점](4-6.md) > [4-6-1. 규칙을 프론트에 새로 만들지 않는다](4-6-1.md)

예를 들어 "이 캐릭터는 이 칸에서 무료" 같은 판단을 프론트에 넣으면 나중에 엔진과 어긋납니다. 프론트는 엔진/백엔드가 준 prompt, choice, view_state를 표현해야 합니다.

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 프론트엔드 관점의 `규칙을 프론트에 새로 만들지 않는다`을 설명합니다. 프론트는 규칙을 판단하는 곳이 아니라 서버가 준 prompt, event, view_state를 사람이 이해할 수 있게 보여주는 계층입니다.

### 관련 코드와 문서

- `apps/web/src/App.tsx`
- `apps/web/src/infra/http/sessionApi.ts`
- `apps/web/src/infra/ws/StreamClient.ts`
- `apps/web/src/hooks/useGameStream.ts`
- `apps/web/src/domain/selectors/`

### 실제 처리 흐름과 기대 값

- 이 문서의 프론트 흐름: 규칙을 프론트에 새로 만들지 않는다.
- 화면에서 시작되는 지점: 보드 위치, 구매 가능 여부, 통행료, 카드 효과를 표시할 때입니다.
- 상태 전달 방식: 서버 `view_state`와 stream event를 selector가 화면용 props로 바꿀 뿐, 컴포넌트가 규칙을 다시 계산하지 않습니다.
- 기대 값/화면 결과: 서버 event와 다른 결과를 만들 수 있는 프론트 계산이 있으면 제거하고, 필요한 값은 백엔드/엔진 payload에 추가해야 합니다.

### 수정할 때 주의할 점

- 프론트에서 새 규칙을 계산하지 말고 prompt/view_state를 믿으세요.
- 화면만 고칠 때도 selector 테스트로 데이터 모델을 먼저 고정하세요.
- sessionId/token/baseUrl이 바뀌면 stream 상태와 중복 decision ledger를 초기화해야 합니다.

### 확인 방법

- `npm --prefix apps/web run test -- --run`
- `npm --prefix apps/web run build`

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
