PromptService는 대기 중인 사람 선택을 저장하고 받습니다. DecisionGateway는 엔진이 요구한 선택을 사람/AI/저장된 결정으로 라우팅합니다.

## 상세 인수인계

### 이 문서에서 이해해야 할 것

이 문서는 백엔드 관점의 `PromptService와 DecisionGateway`을 설명합니다. 백엔드는 API 요청, 엔진 실행, stream 발행, 저장소 연결, visibility projection 사이의 접착층입니다.

### 관련 코드와 문서

- `apps/server/src/app.py`
- `apps/server/src/state.py`
- `apps/server/src/routes/`
- `apps/server/src/services/`
- `apps/server/src/domain/view_state/`

### 실제 처리 흐름과 기대 값

- 이 문서의 흐름: PromptService와 DecisionGateway.
- 진입점: 엔진이 선택을 요청하면 DecisionGateway가 prompt를 발행하고, PromptService가 request_id별 응답을 보관합니다.
- 내부 처리: 프론트 decision은 먼저 ack되고, 런타임이 같은 request_id를 소비해야 실제 게임이 바뀝니다.
- 기대 값/검증 포인트: 기대 값은 `request_id`, `decision_name`, `status=accepted/resolved`, timeout fallback 여부입니다.

### 수정할 때 주의할 점

- 라우터는 얇게, 실제 상태 변경은 서비스에서 찾으세요.
- stream payload에는 semantic guard와 visibility projection 경계가 걸려야 합니다.
- 복구 문제는 runtime-status, replay, Redis store를 함께 확인하세요.

### 확인 방법

- `python3 tools/plan_policy_gate.py`
- 관련 서버/프론트/엔진 테스트를 함께 실행하세요.

### 초보자가 자주 헷갈리는 질문

- “이 값은 프론트가 계산하나요?”: 기본적으로 아니요. 규칙/권한/비공개성은 엔진과 백엔드가 결정해야 합니다.
- “화면이 맞으면 구현도 맞나요?”: 아니요. stream event, view_state, 테스트가 같은 흐름을 말해야 합니다.
- “어디부터 로그를 보나요?”: stream `seq`를 먼저 보고, 프롬프트 문제면 `request_id`, 런타임 문제면 `module_id`를 봅니다.
