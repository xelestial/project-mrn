# 서버 권위 게임 로직 설계안

작성일: 2026-05-12
작성 범위: 현재 구현을 전제로 하지 않는 장기 서버 권위(authoritative server) 설계 제안
근거: 공유된 핵심 게임 룰 요약

## 1. 설계 목표

이 설계의 목표는 프론트엔드, AI, 외부 클라이언트가 어떤 값을 보내더라도 게임의 룰 판정과 상태 변경은 서버가 최종 검증하고 확정하는 구조를 만드는 것이다. 서버는 단순 API 라우터가 아니라 게임 인스턴스별 권위 실행자여야 한다.

핵심 목표:

- 게임 인스턴스 단위로 단일 순서의 명령 처리 보장
- 드래프트, 최종 캐릭터 선택, 턴, 마크 발동, 트릭 카드, 이동, 도착지 처리, 파산, 라운드 종료 보충을 명시적 단계로 모델링
- 외부 입력 프롬프트를 상태 기계의 대기점으로 취급하고, 응답은 항상 서버에서 재검증
- 재시도, 중복 요청, 네트워크 재연결에도 동일한 게임 결과 보장
- 장애 후 스냅샷과 이벤트 로그로 게임을 복구
- 특정 서버 프로세스나 웹소켓 연결에 게임 권위를 묶지 않음

## 2. 비목표

이 문서는 현재 코드 구조에 맞춘 패치 계획이 아니다. 기존 모듈명, 기존 API, 현재 저장소 구조, 프론트엔드 상태 표현을 기준으로 설계를 축소하지 않는다.

비목표:

- 프론트엔드 편의를 위해 룰 검증을 클라이언트에 위임
- 웹소켓 메시지를 곧바로 상태 변경으로 간주
- 플레이어별 UI 상태와 권위 상태를 하나의 모델에 섞기
- AI 응답 형식에 서버 룰을 종속
- 모든 게임을 하나의 전역 락이나 하나의 프로세스 메모리에 묶기
- 로그를 권위 데이터로 사용

## 3. 핵심 모델

### 3.1 GameInstance Actor

각 게임은 하나의 `GameInstance Actor`가 소유한다. 이 Actor는 해당 게임의 명령을 직렬로 처리하고, 상태 변경을 이벤트로 기록한 뒤, 구독자에게 관측 가능한 결과를 발행한다.

Actor 책임:

- 게임 상태의 유일한 쓰기 주체
- 현재 단계와 허용 입력 타입 관리
- 외부 입력의 유효성 검증
- 룰 효과 실행 순서 보장
- 이벤트 로그 append와 스냅샷 생성
- 웹소켓 브로드캐스트용 공개 뷰 생성
- 시간 제한, 재시도, 지연 효과, 실패 효과 처리
- 장애 복구 시 스냅샷과 이벤트 로그로 자기 상태 재구성

Actor가 하지 말아야 할 일:

- 플레이어 UI 레이아웃 결정
- AI 프롬프트 문장 생성에 직접 종속
- 특정 웹소켓 세션을 플레이어의 권위 상태로 간주
- 데이터베이스 트랜잭션 밖에서 확정 상태를 먼저 브로드캐스트

### 3.2 내부 상태

Actor 내부 상태는 적어도 다음 영역으로 분리한다.

- `game`: 게임 ID, 룰 버전, 난수 시드, 현재 라운드, 현재 턴, 현재 단계, 종료 여부
- `players`: 플레이어 ID, 좌석, 연결 상태, 캐릭터 선택 상태, 공개 여부, 자원, 위치, 파산 여부
- `draft`: 드래프트 패스 번호, 후보 카드, 선택 기록, 최종 선택 후보
- `characters`: 캐릭터별 우선순위, 능력, 공개 상태, 소유 플레이어
- `marks`: 예약된 마크 효과, 대상 캐릭터, 예약자, 발동 조건, 만료/실패 정책
- `hands`: 플레이어별 트릭 카드 슬롯 5개, 숨김 슬롯, 사용 가능 여부, 종료 시간, 부담 카드
- `board`: 타일 소유권, 가격, 통행료, 적대 상태, 점수 코인, 특수 효과
- `resources`: 현금, 조각, 점수 토큰, 시작 선택 보상, 랩 보상
- `pendingPrompt`: 현재 외부 입력 요청, 타입, 대상 플레이어, 허용 선택지, 만료 시간, idempotency 키
- `effectQueue`: 도착지, 행운, 연쇄 효과, 마크, 랩 보상 같은 후속 효과 큐
- `audit`: 마지막 처리 명령 ID, 이벤트 시퀀스, 스냅샷 버전

상태 모델의 원칙은 “권위 상태, 비밀 상태, 공개 뷰”를 분리하는 것이다. 예를 들어 손패 5장 중 숨김 카드 1장은 권위 상태에는 존재하지만 다른 플레이어 공개 뷰에는 노출되지 않아야 한다.

## 4. Saga/State Machine 구성

게임 진행은 하나의 거대한 함수가 아니라 작은 Saga들의 연결로 구성한다. 각 Saga는 명확한 시작 조건, 대기 입력, 상태 변경, 완료 이벤트를 가진다.

### 4.1 전체 게임 Saga

상위 상태:

1. `INITIALIZING`
2. `ROUND_START`
3. `DRAFT_PASS_1`
4. `DRAFT_PASS_2`
5. `FINAL_CHARACTER_SELECTION`
6. `TURN_ORDER_RESOLUTION`
7. `PLAYER_TURN`
8. `ROUND_END_REFILL`
9. `GAME_END`

라운드는 날씨 공개로 시작한다. 그 뒤 캐릭터 드래프트 2패스를 진행하고, 최종 캐릭터를 선택한다. 최종 캐릭터의 우선순위가 낮은 플레이어부터 턴을 갖는다.

### 4.2 Draft Saga

입력 프롬프트:

- `draft_card`
- `final_character`

서버 검증:

- 요청 플레이어가 현재 드래프트 선택권자인지 확인
- 선택 카드가 해당 플레이어의 현재 후보군에 있는지 확인
- 같은 패스에서 중복 선택이 아닌지 확인
- 최종 캐릭터가 플레이어의 합법 후보인지 확인

결과:

- 선택 기록 이벤트 저장
- 다음 플레이어 또는 다음 패스로 전환
- 두 패스 완료 후 최종 캐릭터 선택 대기
- 모든 최종 선택 완료 후 턴 순서 계산

### 4.3 Turn Saga

한 플레이어 턴은 다음 하위 단계로 구성한다.

1. `TURN_BEGIN`
2. `RESOLVE_INCOMING_MARKS`
3. `CHARACTER_ABILITY`
4. `OPTIONAL_TRICK`
5. `MOVEMENT_CHOICE`
6. `MOVE_AND_LAND`
7. `RESOLVE_DESTINATION`
8. `RESOLVE_LAP_REWARD`
9. `RESOLVE_TILE_ACTIONS`
10. `RESOLVE_EFFECT_QUEUE`
11. `TURN_END`

각 단계는 완료 이벤트를 남긴다. 단계 중 외부 입력이 필요한 경우 `pendingPrompt`를 만들고 Actor는 대기 상태로 들어간다.

### 4.4 Mark Saga

마크 능력은 플레이어가 아니라 캐릭터를 대상으로 예약된다. 발동 시점은 대상 캐릭터를 선택한 플레이어의 턴 시작이다.

필수 정책:

- 마크 예약 시 대상 캐릭터가 합법 대상인지 검증
- 대상 캐릭터가 아직 공개되지 않은 경우 발동 가능 여부를 별도 상태로 보존
- 대상 캐릭터를 아무도 선택하지 않았으면 실패 또는 만료
- 대상 캐릭터가 공개되었고 소유 플레이어가 있으면 해당 플레이어의 `TURN_BEGIN`에서 발동
- 여러 마크가 같은 턴에 발동하면 서버가 정의한 안정적 순서로 처리

마크는 즉시 상태를 바꾸는 명령이 아니라 미래 조건에 걸린 예약 효과다. 이 차이를 모델에 강제해야 한다.

### 4.5 Trick Card Saga

입력 프롬프트:

- `trick_to_use`
- `active_flip`
- `burden_exchange`

서버 검증:

- 현재 플레이어의 턴인지 확인
- 이번 턴에 이미 트릭 카드를 사용하지 않았는지 확인
- 선택 카드가 플레이어 손패의 공개/사용 가능 슬롯에 있는지 확인
- 숨김 카드의 공개 또는 사용 조건이 충족되었는지 확인
- 부담 카드 제거 비용을 지불할 수 있는지 확인

라운드 종료 보충:

- 종료 시간이 3의 배수인 빈 슬롯만 보충
- 보충은 라운드 종료 Saga에서 일괄 처리
- 보충 결과는 이벤트로 기록하여 재현 가능하게 만든다

### 4.6 Movement and Landing Saga

입력 프롬프트:

- `movement`
- `runaway_step`

서버 검증:

- 주사위 또는 주사위 카드 선택이 현재 상태에서 가능한지 확인
- 이동량은 서버 난수 또는 서버가 검증한 카드 효과에서만 확정
- 경로 중 랩 통과 여부를 서버가 계산
- 도착 타일 효과, 행운, 연쇄 효과는 `effectQueue`로 순서화

이동 결과는 단일 이벤트가 아니라 다음 값을 포함해야 한다.

- 출발 위치
- 이동 수단
- 이동량
- 경유한 주요 지점
- 도착 위치
- 랩 보상 발생 여부
- 후속 효과 큐

### 4.7 Tile and Resource Saga

입력 프롬프트:

- `purchase_tile`
- `lap_reward`
- `coin_placement`
- `geo_bonus`

서버 검증:

- 구매 가능한 타일인지 확인
- 통행료, 인수 비용, 보상, 점수 코인 배치 조건을 서버가 계산
- 현금, 조각, 점수 토큰 변화가 음수 허용 규칙을 위반하지 않는지 확인
- 지불 불능이면 즉시 파산 Saga로 전환

파산 처리:

- 지불 불능 플레이어는 파산 상태가 된다
- 보유지는 적대 상태가 된다
- 이후 해당 보유지 통행료는 은행에 3배로 납부한다
- 파산 이벤트는 자원 변경, 소유권 변경, 타일 상태 변경을 하나의 원자적 결과로 남긴다

## 5. 외부 입력 프롬프트 계약

프롬프트는 “질문”이 아니라 “현재 상태에서 허용되는 명령 슬롯”이다. 서버는 매 프롬프트마다 선택지를 생성하고, 클라이언트나 AI는 그중 하나를 선택해 응답한다.

공통 필드:

```json
{
  "gameId": "game-123",
  "promptId": "prompt-456",
  "type": "movement",
  "targetPlayerId": "player-1",
  "stateVersion": 128,
  "expiresAt": "2026-05-12T12:00:00Z",
  "choices": [],
  "metadata": {}
}
```

응답 공통 필드:

```json
{
  "gameId": "game-123",
  "promptId": "prompt-456",
  "commandId": "client-generated-uuid",
  "playerId": "player-1",
  "choice": {},
  "observedStateVersion": 128
}
```

서버 처리 원칙:

- `promptId`가 현재 대기 프롬프트와 일치해야 한다
- `playerId`가 대상 플레이어와 일치해야 한다
- `commandId`로 중복 응답을 감지한다
- `choice`는 서버가 만든 선택지 또는 서버가 검증 가능한 범위 안에 있어야 한다
- `observedStateVersion`이 오래되어도 자동 거절하지 않는다. 단, 선택이 현재 프롬프트와 불일치하면 거절한다

지원 프롬프트 타입:

- `draft_card`
- `final_character`
- `trick_to_use`
- `movement`
- `purchase_tile`
- `mark_target`
- `active_flip`
- `burden_exchange`
- `lap_reward`
- `runaway_step`
- `coin_placement`
- `geo_bonus`

## 6. 웹소켓 계약

웹소켓은 권위 경로가 아니라 관측과 명령 제출 경로다. 연결이 끊겨도 게임은 계속 복구 가능해야 하며, 재연결한 클라이언트는 서버 스냅샷과 이벤트 시퀀스로 따라잡는다.

### 6.1 클라이언트에서 서버로

메시지:

- `subscribe_game`: 게임 구독
- `unsubscribe_game`: 게임 구독 해제
- `submit_prompt_response`: 프롬프트 응답 제출
- `request_snapshot`: 현재 공개 뷰 요청
- `ack_events`: 받은 이벤트 시퀀스 확인
- `ping`: 연결 상태 확인

`submit_prompt_response`는 곧바로 상태 변경을 뜻하지 않는다. Actor의 명령 큐에 들어가고, Actor가 검증한 뒤 확정 이벤트를 발행한다.

### 6.2 서버에서 클라이언트로

메시지:

- `game_snapshot`: 현재 공개 뷰
- `game_event`: 확정된 상태 변경 이벤트
- `prompt_opened`: 외부 입력 대기 시작
- `prompt_closed`: 입력 수락, 만료, 취소, 실패
- `command_accepted`: 명령 큐 수락
- `command_rejected`: 검증 실패
- `connection_notice`: 재동기화 필요, 백프레셔, 서버 이전 등

각 이벤트에는 단조 증가하는 `eventSeq`를 포함한다.

```json
{
  "gameId": "game-123",
  "eventSeq": 129,
  "stateVersion": 129,
  "type": "player_moved",
  "payload": {},
  "visibility": "public"
}
```

비밀 정보는 플레이어별 뷰에서 분기한다. 같은 `eventSeq`라도 수신자에 따라 payload가 다를 수 있으므로, 내부 권위 이벤트와 외부 송신 이벤트를 분리한다.

## 7. 영속화, 스냅샷, 복구

권위 데이터는 append-only 이벤트 로그와 주기적 스냅샷을 함께 사용한다.

### 7.1 이벤트 로그

저장 대상:

- 명령 수락/거절 결과
- 드래프트 선택
- 최종 캐릭터 선택
- 마크 예약/발동/실패
- 카드 사용/보충/제거
- 이동 결정과 결과
- 자원 변경
- 타일 구매/통행료/인수/적대화
- 파산
- 프롬프트 열림/닫힘
- 라운드 시작/종료
- 게임 종료

이벤트 로그는 재생 가능해야 한다. 서버 난수 결과도 이벤트에 기록한다. “주사위를 굴렸다”가 아니라 “서버 시드 기준 결과가 4였다”까지 확정 이벤트에 남겨야 한다.

### 7.2 스냅샷

스냅샷 생성 시점:

- 라운드 시작
- 라운드 종료
- N개 이벤트마다
- 게임 종료
- Actor 이전 전

스냅샷 내용:

- 전체 권위 상태
- 비밀 상태 포함
- 마지막 적용 이벤트 시퀀스
- 룰 버전
- 난수 상태
- 현재 pendingPrompt
- effectQueue

### 7.3 복구

복구 절차:

1. 최신 스냅샷 로드
2. 스냅샷 이후 이벤트 로그 재생
3. 현재 pendingPrompt와 timeout 재등록
4. 구독자에게 `game_snapshot` 발행
5. 누락 이벤트가 필요한 클라이언트에는 `eventSeq` 기준으로 catch-up 제공

복구 중 중복 실행을 막기 위해 이벤트 append와 Actor 상태 적용은 같은 순서 보장을 가져야 한다. 외부 브로드캐스트는 저장 성공 후에만 한다.

## 8. 동시성, 스케일링, 샤딩, 백프레셔

### 8.1 동시성

게임 인스턴스 내부는 단일 writer 원칙을 따른다. 같은 게임에 들어오는 모든 명령은 Actor mailbox에 직렬화된다. 여러 플레이어가 동시에 응답해도 현재 pendingPrompt와 일치하는 하나만 수락된다.

동시성 원칙:

- 게임 간 병렬성은 허용
- 게임 내부 상태 변경은 직렬 처리
- 읽기 뷰 생성은 스냅샷 기반으로 병렬 가능
- 명령 수락 순서는 서버 수신 순서가 아니라 Actor mailbox 확정 순서

### 8.2 스케일링

게임 수가 늘어나면 GameInstance Actor를 여러 워커에 분산한다. 웹소켓 서버는 어느 노드에 있어도 되고, 명령은 라우터를 통해 해당 게임의 소유 Actor로 전달된다.

필수 구성:

- `gameId` 기반 consistent hashing 또는 shard map
- Actor lease 또는 ownership record
- 워커 장애 시 lease 만료 후 다른 워커가 Actor 복구
- 브로드캐스트는 pub/sub 또는 event fanout 계층 사용

### 8.3 샤딩

샤딩 키는 `gameId`다. 플레이어 ID나 방 ID로 샤딩하면 같은 게임의 명령이 여러 writer로 갈라질 위험이 있다.

샤딩 정책:

- 한 게임은 한 시점에 하나의 Actor만 소유
- 소유권 이전 시 새 명령 처리를 잠시 멈추고 스냅샷을 기준으로 재개
- shard map 변경은 새 게임부터 적용하고, 진행 중 게임은 안정적으로 drain하는 전략을 우선

### 8.4 백프레셔

백프레셔 대상:

- Actor mailbox 길이
- 이벤트 로그 저장 지연
- 웹소켓 송신 큐
- AI/외부 입력 응답 지연
- 스냅샷 생성 지연

정책:

- mailbox가 임계치를 넘으면 새 명령에 `server_busy` 거절 또는 지연 응답
- 느린 웹소켓 클라이언트는 이벤트 전체 전송 대신 스냅샷 재동기화로 전환
- 외부 입력이 만료되면 기본 정책 또는 실패 정책을 Saga가 결정
- 저장소 지연이 발생하면 확정 이벤트 발행을 멈춘다. 저장되지 않은 상태는 권위 상태가 아니다

## 9. 실패 처리와 재시도/중복 처리

### 9.1 Idempotency

모든 클라이언트 명령은 `commandId`를 가진다. Actor는 게임별로 처리된 `commandId`와 결과를 보관한다.

동일 `commandId` 재수신:

- 이미 수락된 명령이면 기존 결과 반환
- 이미 거절된 명령이면 기존 거절 사유 반환
- 처리 중이면 `in_progress` 반환

### 9.2 Prompt 만료

프롬프트는 타입별 만료 정책을 가진다.

- 필수 선택: 기본 선택 또는 자동 실패 정책 필요
- 선택적 트릭 카드: 미사용으로 진행 가능
- 구매 선택: 구매하지 않음으로 진행 가능
- 마크 대상 선택: 예약 실패 또는 기본 대상 없음
- 코인 배치 등 전략 선택: 규칙상 기본값이 없으면 명확한 실패 상태로 멈춤

만료도 이벤트로 기록한다.

### 9.3 외부 서비스 실패

AI나 클라이언트 입력 실패는 게임 서버 실패가 아니다. 서버는 프롬프트 만료와 대체 정책으로 진행하거나, 사람이 개입해야 하는 `PAUSED_FOR_INPUT` 상태로 전환한다.

### 9.4 저장소 실패

이벤트 로그 append 실패 시 상태를 확정하지 않는다. Actor는 명령을 재시도 가능 상태로 유지하거나 게임을 `PAUSED_DURABILITY_ERROR`로 전환한다.

### 9.5 중간 브로드캐스트 실패

브로드캐스트 실패는 권위 상태를 되돌릴 이유가 아니다. 클라이언트는 재연결 후 스냅샷과 이벤트 시퀀스로 복구한다.

## 10. 운영 관측성

운영 지표는 “서버가 살아 있다”보다 “게임 진행이 멈추지 않는다”를 중심으로 잡아야 한다.

필수 메트릭:

- 활성 게임 수
- Actor mailbox 길이
- 명령 처리 지연 p50/p95/p99
- 프롬프트 대기 시간
- 프롬프트 만료율
- 이벤트 로그 append 지연
- 스냅샷 생성 시간과 크기
- 복구 횟수와 복구 시간
- websocket fanout 지연
- command rejected 사유별 카운트
- 파산, 마크 실패, 부담 카드 제거 같은 룰 이벤트 카운트

필수 로그 필드:

- `gameId`
- `actorId`
- `shardId`
- `eventSeq`
- `stateVersion`
- `commandId`
- `promptId`
- `playerId`
- `saga`
- `step`
- `ruleVersion`

트레이싱:

- 명령 수신부터 이벤트 append, Actor 적용, websocket 발행까지 하나의 trace로 연결
- 외부 입력 프롬프트는 열림부터 응답, 검증, 닫힘까지 span으로 기록

알람:

- 특정 게임의 pendingPrompt가 과도하게 오래 유지
- Actor mailbox가 지속 증가
- 이벤트 로그 append 실패
- 복구 반복
- websocket 송신 큐 누적
- 같은 commandId의 비정상 재시도 급증

## 11. 단계별 이전 전략

현재 구현과 무관하게, 안전한 이전은 권위 경계를 먼저 세우는 순서로 진행해야 한다.

### 1단계: 룰 이벤트 정의

권위 이벤트 타입과 상태 전이 표를 먼저 정의한다. 이 단계에서 프론트엔드 메시지나 기존 API 이름에 맞추지 않는다.

산출물:

- 게임 상태 스키마
- 이벤트 타입 목록
- 프롬프트 타입별 선택지 스키마
- Saga 단계 표

### 2단계: 단일 프로세스 Actor

하나의 프로세스 안에서 GameInstance Actor를 구현한다. 저장소와 웹소켓은 단순해도 되지만, 게임 내부 명령은 반드시 mailbox를 통해 직렬 처리한다.

검증:

- 4인 드래프트부터 1라운드 종료까지 재현 가능한 시뮬레이션
- 동일 commandId 재시도 결과 동일성
- 잘못된 prompt 응답 거절

### 3단계: 이벤트 로그와 스냅샷

상태 변경을 이벤트 로그에 append한 뒤 적용하도록 바꾼다. 스냅샷 재생으로 동일 상태가 나와야 한다.

검증:

- 임의 시점 스냅샷 후 재생
- Actor 강제 종료 후 복구
- pendingPrompt 복구

### 4단계: 웹소켓 공개 뷰 분리

권위 이벤트에서 플레이어별 공개 뷰를 생성한다. 비밀 카드와 비공개 캐릭터 정보가 다른 플레이어에게 새지 않는지 검증한다.

### 5단계: 분산 Actor 소유권

`gameId` 기반 샤딩과 Actor lease를 추가한다. 웹소켓 노드와 Actor 노드를 분리한다.

검증:

- 워커 장애 후 lease 이전
- 진행 중 게임 복구
- 느린 클라이언트 재동기화

### 6단계: 운영 자동화

관측성, 알람, 백프레셔 정책을 운영 기본값으로 고정한다. 문제를 로그 파일 분석으로만 찾는 구조는 장기적으로 실패한다.

## 12. 거부해야 할 나쁜 결합

다음 결합은 명확히 거부해야 한다.

- 프론트엔드 상태를 권위 상태로 간주
- 웹소켓 연결 세션에 현재 턴 권한을 저장
- AI 응답을 검증 없이 상태 변경으로 적용
- prompt text를 파싱해서 룰을 결정
- 카드 이름 문자열 비교로 효과 실행
- 랜덤 결과를 이벤트에 남기지 않고 재실행 시 다시 굴림
- 라운드/턴 진행을 타이머 콜백 여러 개에 분산
- 마크 예약 효과를 대상 플레이어에 즉시 붙이고 캐릭터 공개/선택 조건을 잊음
- 손패 숨김 정보와 공개 뷰를 같은 객체로 공유
- 저장소 append 전에 websocket으로 확정 이벤트 발행
- 모든 게임을 하나의 전역 큐로 처리
- 게임별 Actor 소유권 없이 여러 서버가 같은 게임 상태를 쓰기
- 복구 로직이 스냅샷 없이 현재 메모리 상태에 의존
- 운영 로그를 비즈니스 이벤트의 대체물로 사용

## 13. 결론

이 게임은 단순 턴제 보드게임처럼 보여도, 드래프트 2패스, 캐릭터 우선순위 턴 순서, 캐릭터 대상 마크 예약, 숨김 손패, 라운드 종료 보충, 연쇄 도착 효과, 파산 후 적대 보유지 같은 지연 효과와 비공개 정보가 많다. 따라서 서버는 API 요청을 받아 상태를 조금 수정하는 CRUD 구조가 아니라, 게임 인스턴스별 Actor가 Saga를 순서대로 실행하는 권위 런타임이어야 한다.

장기적으로 견고한 구조는 다음 원칙을 지켜야 한다.

- 한 게임에는 한 writer만 둔다.
- 모든 외부 입력은 promptId와 선택지 기준으로 검증한다.
- 모든 확정 상태 변경은 이벤트로 남긴다.
- 스냅샷과 이벤트 재생으로 복구한다.
- 웹소켓은 관측 경로이며 권위 경로가 아니다.
- 비밀 상태와 공개 뷰를 분리한다.
- 룰 효과는 Saga와 effectQueue로 순서화한다.

이 원칙을 지키면 현재 구현 방식과 무관하게 서버 권위를 유지할 수 있고, AI 자동 플레이, 사람 플레이, 재연결, 장애 복구, 분산 실행을 같은 모델 안에서 다룰 수 있다.
