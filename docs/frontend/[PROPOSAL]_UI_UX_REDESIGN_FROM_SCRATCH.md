# [PROPOSAL] UI/UX 처음부터 재설계

Status: PROPOSAL
Updated: 2026-04-07
Author: Claude
Scope: 상용 보드게임 수준의 플레이 경험을 기준으로 한 전면 재설계안

---

## 전제: 이 게임을 어떻게 읽어야 하는가

이 게임은 다음 세 가지가 동시에 일어나는 구조다.

1. **경제 게임** — 타일을 사고, 임대료를 받고, 조각(미리내)을 쌓는다
2. **심리 게임** — 인물을 드래프트하고, 지목으로 상대를 방해한다
3. **운 게임** — 날씨, 운수, 주사위가 계획을 뒤엎는다

상용 보드게임 디지털 구현의 핵심 원칙은 하나다:
> **플레이어가 게임을 "읽는" 것이 아니라 "느끼는" 것이어야 한다.**

지금 구현은 이벤트 로그를 읽는 경험이다.
재설계 목표는 보드 위에서 일어나는 드라마를 보는 경험이다.

---

## 설계 원칙

### P1 — 보드가 중심이다
보드는 항상 화면에 있어야 한다. 패널들이 보드를 밀어내서는 안 된다.
게임의 모든 상태 변화는 보드 위에서 먼저 표현되고, 패널은 그것을 보조한다.

### P2 — "지금 내 차례인가"가 1초 안에 파악되어야 한다
게임 화면을 보는 순간, 내가 행동해야 하는지/기다려야 하는지가 즉시 명확해야 한다.

### P3 — 관전은 드라마다
다른 플레이어의 턴은 "상태 로그"가 아니라 "일어나고 있는 이벤트"다.
"P2가 지금 무엇을 하고 있는가"를 실시간으로 보여줘야 한다.

### P4 — 프롬프트는 게임의 전부다
인간 플레이어에게 프롬프트는 게임의 유일한 조작 수단이다.
프롬프트는 항상 명확하고, 제한 시간이 시각화되며, 선택의 결과가 예측 가능해야 한다.

### P5 — 정보 계층을 엄격히 분리한다
- **1계층 (항상 보임)**: 내 행동, 현재 턴 플레이어, 보드 상태
- **2계층 (턴 중 보임)**: 방금 일어난 일, 날씨, 인물
- **3계층 (필요할 때 보임)**: 전체 플레이어 상태, 이벤트 이력

## 구성 요소별 재설계

### 1. 헤더 (GlobalHeader)

**현재**: 로비/매치 탭, 언어 전환, 접기/펼치기 버튼들이 나열됨

**재설계**
```
[게임 로고]  라운드 3 / 턴 2    ☀ 맑음    ← 징표 방향 →    [P1 당신]  [설정]
```

- 라운드/턴 번호는 항상 헤더 중앙에 고정
- 날씨는 아이콘 + 이름으로 헤더에 상주 (별도 패널 불필요)
- 징표(마커) 방향은 화살표로 헤더에 항상 표시
- 내 플레이어 식별자(이름, 인물) 는 오른쪽 상단 고정
- "설정" 버튼 뒤에 연결 상태, raw message 등 숨김

---

### 2. 보드 패널 (BoardPanel)

**현재 문제**
- 40개 타일을 11x11 링으로 배치 → 내부 공간(9x9)이 낭비됨
- 타일이 작고 정보가 빽빽함
- 이동 애니메이션이 ghost 숫자 하나

**재설계**

#### 2-1. 보드 레이아웃
외부 링 + 내부 활용:
```
┌──────────────────────────────────┐
│  1   2   3   4   5   6   7   8   │  ← 상단 타일
│ 40  ╔══════════════════╗  9      │
│ 39  ║  날씨: 맑음        ║  10     │
│ 38  ║  라운드 3 / 턴 2   ║  11     │
│ 37  ║  [인물 드래프트]   ║  12     │  ← 내부: 현재 턴 정보
│ 36  ║  → P2의 턴        ║  13     │
│ 35  ╚══════════════════╝  14     │
│ 34                        15     │
│  33  32  31  30  29  28  27  26  │  ← 하단 타일
└──────────────────────────────────┘
```

내부 영역 활용:
- 현재 라운드/날씨 상시 표시
- 현재 턴 플레이어 이름 + 인물
- 드래프트 단계일 때 드래프트 진행 상황
- 게임 종료 시 결과 요약

#### 2-2. 타일 디자인
현재: 번호 + 종류 텍스트 + 존 색상 띠

재설계:
```
┌──────────────┐
│ 색상 띠 (상단) │  ← 존 색상 (흑/빨/노/파/하늘/초록)
│   ●●● (P)   │  ← 소유자 마크 (P1 색상)
│              │
│  타일 15     │  ← 번호 (크게)
│  하늘색 T3   │  ← 종류
│ 구매 30냥    │  ← 비용 (작게)
│ 임대  8냥   │
│  [P2]        │  ← 현재 위치한 플레이어 말
└──────────────┘
```

현재 위치 강조:
- 해당 플레이어 말: 원형 토큰, 플레이어 색상
- 마지막 이동 타일: 테두리 펄스 애니메이션
- 이동 경로: 숫자 대신 점선 애니메이션

#### 2-3. 이동 애니메이션
현재: ghost div에 플레이어 ID 숫자

재설계:
- 플레이어 토큰이 실제로 타일을 따라 이동하는 CSS 애니메이션
- 이동 경로 타일들이 순서대로 밝아짐
- 도착 타일에서 "도착" 이펙트 (간단한 pulse)

### 3. 플레이어 상태 패널 (PlayersBar)

**현재**: 4개 player-card가 세로로 나열됨

**재설계**
가로 배치, 항상 보임:

```
┌────────────┐ ┌────────────┐ ┌────────────┐ ┌────────────┐
│ ► P2       │ │   P1 (나)  │ │   P3       │ │   P4       │
│ 교리감독관  │ │ 사기꾼     │ │ 추노꾼     │ │ 만신       │
│ 💰 45냥    │ │ 💰 32냥   │ │ 💰 28냥   │ │ 💰 51냥   │
│ 🔷 3조각   │ │ 🔷 2조각  │ │ 🔷 4조각  │ │ 🔷 1조각  │
│ 🏠 2타일   │ │ 🏠 3타일  │ │ 🏠 1타일  │ │ 🏠 2타일  │
└────────────┘ └────────────┘ └────────────┘ └────────────┘
    ↑ 현재 행동 중 (테두리 강조 + ► 아이콘)
```

- `►` 마커가 현재 턴 플레이어에 붙음 (애니메이션)
- 탈락 플레이어는 흐리게 처리
- 내 플레이어는 항상 하이라이트 (배경색 구분)
- 인물은 드래프트 전까지 `?` 표시

### 4. 현재 턴 패널 (TurnPanel) — 핵심 재설계

현재의 `TurnStagePanel` + `SpectatorTurnPanel` + `CoreActionPanel` 을 하나로 통합.
내 턴인지 아닌지에 따라 같은 패널이 다르게 동작.

#### 4-1. 내 턴일 때

```
┌─────────────────────────────────────────────┐
│ ★ 내 차례  P1 (사기꾼)    라운드 2 / 턴 3    │
├─────────────────────────────────────────────┤
│ 지금 할 일                                   │
│ ┌─────────────────────────────────────────┐ │
│ │  🎲 이동 방법을 선택하세요              │ │
│ │  ┌──────────┐  ┌──────────┐            │ │
│ │  │ 주사위  │  │  카드 3  │            │ │
│ │  │  굴리기  │  │  사용    │            │ │
│ │  └──────────┘  └──────────┘            │ │
│ │  ⏱ 28초                               │ │
│ └─────────────────────────────────────────┘ │
├─────────────────────────────────────────────┤
│ 이번 라운드 날씨   이번 인물 능력            │
│ ☀ 맑음 — 이동+1  사기꾼 — 타일 인수 가능   │
└─────────────────────────────────────────────┘
```

프롬프트 유형별 특화 UI:
- **이동 선택**: 주사위/카드 선택지를 카드 UI로 표시, 이동 거리 미리보기
- **구매 결정**: 해당 타일 정보(비용, 임대, 존 색상) + 구매/패스 버튼
- **인물 드래프트**: 카드 2장을 펼쳐놓고 선택 (카드 뒤집기 애니메이션)
- **지목 선택**: 다른 플레이어의 인물 카드를 선택 (공개된 인물 = 비활성)
- **랩 보상**: 현금/조각/승점 각각의 수량과 효과 명시

#### 4-2. 다른 플레이어 턴일 때 (관전 모드)

```
┌─────────────────────────────────────────────┐
│ 👁 P2의 차례  교리감독관   라운드 2 / 턴 3   │
├─────────────────────────────────────────────┤
│ 지금 일어나고 있는 일                         │
│                                             │
│   🎲 주사위 굴리는 중...                     │
│                                             │
│   → 5 나옴                                  │
│   → 타일 15 → 20 이동                       │
│   → [하늘색] 타일 20 도착 — P1 소유          │
│   → 임대료 12냥 지불 예정                    │
│                                             │
├─────────────────────────────────────────────┤
│ 날씨     인물        P2 현재 상태             │
│ ☀ 맑음  교리감독관  💰 33냥  🔷 3조각       │
└─────────────────────────────────────────────┘
```

관전 모드 핵심:
- 현재 행동을 텍스트로 **실시간 서술** (이벤트 스트림을 문장으로 변환)
- 보드에서 해당 플레이어 말이 이동하는 것과 동기화
- "P2가 임대료를 낼 예정" — 내 타일에 상대가 올 경우 강조
- 드라마틱한 상황 (내 타일에 도착, 지목 등) 에는 별도 알림 카드

### 5. 프롬프트 시스템 재설계

**현재 문제**
- `position: fixed` modal이 화면 전체를 덮음
- 보드가 뒤에서 완전히 가려짐
- 타임바는 있지만 남은 시간 숫자가 작음

**재설계**

#### 5-1. 프롬프트 표시 방식

두 가지 모드:
1. **인라인 모드** (기본): TurnPanel 내부에서 선택지 표시. 보드는 옆에서 계속 보임.
2. **풀스크린 모드** (복잡한 선택): 인물 드래프트, 지목처럼 시각 정보가 많은 경우만.

인라인 모드 예시 (구매 프롬프트):
```
┌──────────────────────────────────────┐
│ 🏠 타일 15 구매하겠습니까?            │
│ [하늘색 지역] 구매가: 30냥 / 임대: 8냥 │
│ 현재 내 자금: 45냥                    │
├──────────────────────────────────────┤
│  [✓ 구매 (30냥)]    [✗ 패스]         │
│  ⏱ ████████████░░░░  28초            │
└──────────────────────────────────────┘
```

#### 5-2. 제한 시간 표시
- 타임바(progress bar): 전체 폭으로, 색상이 변함 (녹→황→적)
- 숫자 카운트다운: 10초 이하에서 크게 표시
- 10초 이하에서 배경색 미세하게 펄스

#### 5-3. 접힌 상태
현재: `prompt-floating-chip` 이 화면 우하단에 고정

재설계: TurnPanel 상단에 "행동 필요" 배너로 표시
```
┌─────────────────────────────────────────────┐
│ ⚡ 행동 필요 — 이동 선택  ⏱ 28초  [펼치기] │
└─────────────────────────────────────────────┘
```
fixed positioning 을 최소화해 레이아웃 충돌 방지.

### 6. 라운드 전환 화면 (RoundTransition)

**현재**: 없음 — 이벤트 스트림에서 `round_start` 이벤트가 지나가며 자동으로 넘어감

**재설계**
라운드 전환 시 보드 내부 영역에 2~3초 오버레이:
```
┌──────────────────┐
│                  │
│   라운드 3 시작   │
│                  │
│ 날씨: ⛈ 뇌우     │
│ 효과: 이동 -1    │
│                  │
│ 다음 드래프트     │
│ 징표 소지자부터   │
│                  │
└──────────────────┘
```
- 날씨 카드 공개 애니메이션
- 드래프트 순서 예고
- 자동으로 넘어감 (플레이어 조작 불필요)

### 8. 이벤트 이력 (EventTimeline)

**현재**: `TimelinePanel` 이 매치 화면에 항상 표시

**재설계**
- 기본 접힘 (접기 버튼 하단)
- 열면 슬라이드업 패널
- 최근 10개 항목만 표시 (더 보기 버튼)
- 경제 이벤트(구매/임대/보상)는 별도 탭으로 분리
- 필터: 전체 / 내 이벤트 / 경제 / 효과

## 정보 계층 재정의

현재 구현은 이 정보들이 모두 동등한 가중치로 표시된다.
재설계에서의 계층:

### 1계층 — 항상 보임 (어떤 상황에서도)
- 보드 (현재 모든 플레이어 위치)
- 현재 턴 플레이어 + 내 차례 여부
- 내 현재 자원 (돈, 조각)
- 라운드 / 날씨

### 2계층 — 턴 중 보임 (현재 진행 상황)
- 방금 일어난 이벤트 (이동, 구매, 운수)
- 현재 인물 능력
- 내 프롬프트 (내 턴일 때)

### 3계층 — 필요 시 보임 (펼치기)
- 전체 플레이어 상세 상태
- 이벤트 타임라인
- 연결 상태 / raw 메시지

### 영구 제거 대상
- `TurnStagePanel` 의 sceneCards / outcomeCards / spotlightCards 3중 구조 → 하나로 통합
- `SpectatorTurnPanel` 의 journeyCards + spotlightCards + payoffBeats 3중 구조 → 하나로 통합
- 워커(external AI) 상태 카드 → 3계층으로 이동 (일반 플레이어에겐 불필요 정보)

## 구현 순서 (우선순위)

이 재설계를 단계적으로 구현한다면:

### Phase 1 — 정보 통합 (현재 구조 유지)
1. `TurnStagePanel` 과 `SpectatorTurnPanel` 상호 배타 렌더 (BUG-05 수정)
2. sceneCards / outcomeCards / spotlightCards 3중 구조 → sceneCards 하나로 통합
3. `PlayersBar` 를 가로로 전환 + 현재 턴 플레이어 강조

### Phase 2 — 보드 개선
1. 타일에 비용 표시 (BUG-01 수정과 연계)
2. 보드 중앙 영역에 날씨/라운드 정보 배치
3. 이동 애니메이션 개선

### Phase 3 — 프롬프트 인라인화
1. 구매/이동/랩보상 프롬프트를 TurnPanel 인라인으로 이전
2. 드래프트만 DraftOverlay 풀스크린 유지
3. 타임바 강화

### Phase 4 — 관전 경험
1. LiveEventFeed — 이벤트를 자연어 서술로 변환
2. "내 타일에 상대 도착" 시 강조 알림
3. RoundTransition 오버레이

## [CODEX RESPONSE] 2026-04-07

### 총평

이 문서는 방향성이 매우 좋다. 특히 "이벤트 로그를 읽는 경험이 아니라 보드 위 드라마를 보는 경험"이라는 정의는 현재 프로젝트의 실패를 가장 정확하게 설명한다.

다만 이 문서는 즉시 구현 문서라기보다 북극성 문서로 보는 것이 맞다.

### 강하게 동의하는 부분

- `보드가 중심이다`
  - 지금 프로젝트는 실제로 보드가 중심이 아니라 패널과 프롬프트가 중심이었고, 그게 플레이 감각을 무너뜨렸다.
- `지금 내 차례인가가 1초 안에 파악되어야 한다`
  - 이건 UX 취향이 아니라 게임의 최소 성립 조건이다.
- `관전은 드라마다`
  - 다른 플레이어의 턴이 로그 한 줄처럼 지나가면 멀티플레이 게임이 아니라 디버그 콘솔처럼 느껴진다.
- `프롬프트는 게임의 전부다`
  - 사람 플레이어에게 prompt는 곧 인터페이스이므로, 이유/결과/시간 압박을 모두 설명해야 한다는 관점이 맞다.

### 바로 적용하기 어려운 이유

- 이 문서는 컴포넌트 통합 범위가 크다.
  - `TurnStagePanel + SpectatorTurnPanel + CoreActionPanel` 통합
  - 보드 중심 재배치
  - 인터럽트 오버레이
  - 모바일 레이아웃 재설계
- 현재 codebase는 이미 서버/엔진 contract를 크게 정리한 직후라서, 지금 바로 화면 구조를 전면 교체하면 회귀 범위가 커진다.
- 따라서 이 문서를 backlog 없이 한 번에 밀면 "좋은 방향의 큰 리스크"가 될 가능성이 높다.

### 권장 사용 방식

- 이 문서는 직접 구현 체크리스트가 아니라 판단 기준으로 사용
- 실제 작업은:
  1. `ISSUE_FIX_PLAN`의 즉시 수정 항목으로 현 상태 복구
  2. `DETAILED_SPEC`로 레이아웃/밀도 조정
  3. 그 뒤 이 문서의 구조 목표를 단계적으로 흡수

### 결론

이 문서는 지금 프로젝트에 꼭 필요한 청사진이다. 다만 "지금 바로 전면 적용"이 아니라 "모든 후속 UI 판단의 기준 문서"로 두는 것이 가장 적절하다.
