# CHANGE_WORKFLOW_GUIDE.md

## 목적
이 문서는 이 프로젝트에서 **작은 수정이든 큰 수정이든 동일한 방식으로 변경 의도와 절차를 기록**하기 위한 내부 지침서다.

핵심 목표는 다음과 같다.
- 왜 수정했는지 잊지 않기
- 어디를 바꿨는지 빠르게 추적하기
- 규칙 변경과 구현 변경을 섞어 적지 않기
- 회귀 분석과 밸런스 비교를 다시 할 수 있게 남기기
- 문서와 코드의 불일치를 줄이기

---

## 언제 기록해야 하는가
다음 중 하나라도 해당하면 반드시 기록한다.

- 규칙이 바뀜
- AI 휴리스틱/우선순위/가중치가 바뀜
- 종료 조건, 점수 계산, 자원 흐름이 바뀜
- 실행 속도/메모리/배치 실행 방식이 바뀜
- 버그 수정이 들어감
- 테스트가 추가/삭제/변경됨
- 문서 구조나 배포 구조가 바뀜

아주 작은 수정이라도 **"왜 바꿨는지"가 남아야 하면 기록 대상**이다.

---

## 무엇을 어디에 기록할 것인가

### 1) 항상 남기는 곳

#### `CHANGELOG.md`
사용자/운영자 관점의 릴리스 단위 요약을 남긴다.

적어야 하는 것:
- 버전
- 날짜
- 큰 변경 요약
- 중요한 수정/버그 수정
- 실행/테스트/배포 관점에서 의미 있는 변경

여기에는 세세한 실험 메모를 길게 쓰지 않는다.

#### 대응 문서 `*.md`
실제로 바꾼 소스가 있다면 **대응하는 md 파일도 반드시 수정**한다.
예:
- `ai_policy.py` 수정 → `ai_policy.md`
- `engine.py` 수정 → `engine.md`
- `simulate_with_logs.py` 수정 → `simulate_with_logs.md`

적어야 하는 것:
- 바뀐 동작
- 의도
- 주의점
- 회귀 위험
- 관련 설정/출력 변화

### 2) 필요 시 추가로 남기는 곳

#### 분석/실험 보고서 `*.md`
아래 경우에는 별도 보고서를 만든다.
- 100게임 이상 배치 비교
- 승률/종료원인/자원흐름 분석
- 특정 프로파일 튜닝 실험
- 파산, 리더 견제, 날씨 영향 등 깊은 분석

파일명 권장 예:
- `v738_control_finisher_analysis.md`
- `alive_bankrupt_sample_report.md`
- `leader_denial_report.md`

#### 구조/프로세스 지침 문서
프로젝트 공통 작업 규칙이 바뀌면 이 문서처럼 별도 가이드 문서를 추가한다.

## 변경 기록 작성 순서
변경은 아래 순서로 진행하는 것을 기본으로 한다.

### A. 변경 전
1. 수정 목적을 한 줄로 적는다.
2. 영향 범위를 적는다.
3. 성공 조건을 적는다.
4. 실패 시 되돌릴 기준을 적는다.

권장 기록 위치:
- 작은 수정: 대응 md 하단 메모
- 큰 수정: 별도 분석/작업 md

### B. 구현 중
1. 실제 수정 파일 목록 기록
2. 의도한 로직과 실제 구현 차이 기록
3. 임시 타협/제외한 항목 기록

### C. 구현 후
1. 테스트 결과 기록
2. 스모크 실행 결과 기록
3. 100게임 등 실험 결과 기록
4. 다음 액션 기록

## 필수 기록 항목
작은 수정/큰 수정 공통으로 아래 항목을 남긴다.

### 최소 필수 항목
- 변경 제목
- 날짜
- 변경 이유
- 수정 파일 목록
- 의도한 동작
- 실제 확인 결과
- 테스트 결과
- 남은 리스크

### 권장 추가 항목
- 영향받는 규칙/프로파일
- 되돌림 포인트
- 실험 기준 버전
- 비교 대상 버전
- 대표 지표
  - 승률
  - 종료 원인
  - 평균 턴수
  - 파산율
  - 랩 보상 분포

## 변경 종류별 작성 기준

### 1) 룰 변경
반드시 적을 것:
- 바뀐 룰 원문
- 바뀐 이유
- 적용 범위
- 예외 처리
- 로그/summary에 드러나는 변화

예:
- 날씨를 라운드 시작 공용 효과로 변경
- 9타일 종료 조건 재도입

### 2) AI 정책 변경
반드시 적을 것:
- 대상 프로파일
- 어떤 상황에서 우선순위가 바뀌는지
- 이전보다 더 선호/덜 선호하는 인물/자원
- 기대 효과
- 실패 가능성

예:
- `control`이 리더 긴급 시 조각 우선
- non-v1이 감독관/연구관으로 리더 필요 face 제거 선호

### 3) 버그 수정
반드시 적을 것:
- 재현 조건
- 원인
- 수정 방식
- 회귀 테스트 유무

예:
- `scammer` NameError
- 문서 무결성 타임스탬프 불일치

### 4) 성능/리팩토링
반드시 적을 것:
- 외부 동작이 바뀌는지 여부
- 캐시/구조 변경 포인트
- 성능 측정 전후 값
- 안전성 검증 방식

예:
- CSV/보드/덱 캐시
- summary-only 경로 정리

## 금지/주의 사항
- 코드만 바꾸고 대응 md를 안 바꾸지 말 것
- CHANGELOG에 실험 잡담을 길게 적지 말 것
- 실험 결과와 확정 결과를 구분 없이 적지 말 것
- 버전 표기와 zip 이름을 다르게 두지 말 것
- 테스트하지 않은 변경을 "정상"이라고 적지 말 것

---

## 권장 운영 규칙
- 작은 수정도 최소 템플릿으로 남긴다.
- 큰 수정은 **별도 분석 보고서 + CHANGELOG + 대응 md** 3군데에 남긴다.
- 배치 실험을 돌렸다면 결과 파일 경로를 함께 적는다.
- 규칙 변경이면 가능한 한 **예시 입력/예상 결과**를 한 줄이라도 적는다.
- 배포 직전에는 아래를 확인한다.
  - `pytest -q`
  - `python test_doc_integrity.py`
  - smoke 실행
  - 버전/zip 이름 일치

## External data changes
When board rules change, update the external board spec files together: `board_layout.json`, `board_layout.csv`, and `board_layout_meta.json` when CSV sidecars are used. Record which board-level metadata changed and why.
