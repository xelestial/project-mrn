# GPT Trick Card Runtime Guide

이 문서는 현재 `GPT/` 소스코드 기준으로 `잔꾀`가 어떻게 정의되고 처리되는지 사람이 읽을 수 있게 정리한 설명서입니다.

기준 소스:
- `GPT/trick.csv`
- `GPT/trick_cards.py`
- `GPT/state.py`
- `GPT/engine.py`
- `GPT/effect_handlers.py`

## 한눈에 보는 구조

현재 잔꾀 시스템은 대략 이렇게 동작합니다.

1. `trick.csv`에서 카드 정의를 읽는다.
2. `trick_cards.py`가 이를 `TrickCardDef`, `TrickCard`로 만든다.
3. `state.py`의 `PlayerState.trick_hand`가 실제 손패를 가진다.
4. `engine.py`가 턴 중 잔꾀 사용 단계와 공개/비공개 상태를 관리한다.
5. `effect_handlers.py`의 `handle_trick_card(...)`가 카드별 실제 효과를 적용한다.

## 카드 데이터 정의

현재 덱 정의는 `GPT/trick.csv`에서 읽습니다.

- 카드 종류 수: `28`
- 각 행은 `전체 장수`, `이름`, `설명`을 가집니다.
- `build_trick_deck()`는 장수만큼 복제된 실제 덱을 만듭니다.
- 각 실카드에는 `deck_index`가 부여됩니다.

## `TrickCard`가 소스코드에서 가지는 핵심 속성

`GPT/trick_cards.py` 기준:

- `name`
- `description`
- `deck_index`

추가 판정:

- `is_burden`
  - 현재 코드상 `무거운 짐`, `가벼운 짐`만 burden으로 취급합니다.
- `burden_cost`
  - `무거운 짐 = 4`
  - `가벼운 짐 = 2`
- `is_anytime`
  - 설명 문자열 안에 `언제나 사용할 수 있습니다`가 있는지로 판정합니다.

중요:
- 즉 `언제나` 여부는 별도 컬럼이 아니라 현재는 설명 문자열 기반입니다.
- 따라서 장기적으로는 별도 구조화된 메타데이터로 분리하는 편이 더 안전합니다.

## 공개 / 비공개 처리

`GPT/state.py` 기준:

- 플레이어는 `trick_hand` 전체를 손에 들고 있습니다.
- 그중 `hidden_trick_deck_index` 1장만 비공개로 숨깁니다.
- `public_trick_cards()`는 숨겨진 1장을 제외한 나머지를 돌려줍니다.
- `hidden_trick_count()`는 손패가 있으면 항상 `1`을 반환합니다.

즉 현재 공개 규칙은:

- 손패가 0장: 비공개 0장
- 손패가 1장 이상: 비공개 1장, 나머지는 공개

## 턴 중 잔꾀 사용 순서

`GPT/engine.py` 기준:

### 1. `언제나` 잔꾀
- 자신의 턴 잔꾀 단계에서 먼저 처리합니다.
- 여러 장을 연속으로 사용할 수 있습니다.
- 엔진은 더 이상 사용할 카드가 없을 때까지 반복합니다.

### 2. 일반 잔꾀
- 그 다음 일반 잔꾀를 처리합니다.
- 일반 잔꾀는 자신의 턴에 기본적으로 `1장만` 사용합니다.

### 3. 예외: 일반 잔꾀 단계에서 직접 쓰지 않는 카드
`engine._is_trick_phase_usable(...)` 기준으로 아래 카드는 일반 잔꾀 단계에서 바로 쓰지 않습니다.

- `강제 매각`
- `뭘리권`
- `뭔칙휜`
- `호객꾼`

즉 이 카드들은 손패에 있어도 기본 잔꾀 단계에선 `held/보류` 성격으로 남습니다.

## 사용 로그

잔꾀를 쓰면 `engine.py`는 `trick_used` 이벤트를 남깁니다.

대표 필드:
- `player`
- `phase`
  - `anytime`
  - `regular`
- `character`
- `card.deck_index`
- `card.name`
- `resolution`
- `decision`

따라서 시각화/리플레이에서도 잔꾀 사용 자체는 이벤트 단위로 추적 가능합니다.

## 보급과 짐 교환

현재 `F` 변화에 따라 보급이 발생하면 `engine._run_supply(...)`가 실행됩니다.

동작:
- burden 카드가 있으면 `choose_burden_exchange_on_supply(...)`로 교환 여부를 묻습니다.
- 비용을 내고 burden을 버린 뒤 새 잔꾀를 1장 뽑습니다.
- 이후 손패가 5장이 되도록 추가로 보충합니다.
- 결과를 `trick_supply` 이벤트로 기록합니다.

즉 burden 처리는:
- 평소 사용해서 버릴 수도 있고
- 보급 때 돈을 내고 교환할 수도 있습니다.

## 현재 실제 효과 처리 방식

잔꾀의 실제 효과는 `GPT/effect_handlers.py`의 `handle_trick_card(...)`에서 처리됩니다.

아래 표는 현재 구현된 처리 기준입니다.

| 카드 | 현재 처리 |
|---|---|
| 성물 수집가 | 이번 턴 조각 추가 획득 버프 |
| 무거운 짐 | 4냥 지불 후 제거 |
| 가벼운 짐 | 2냥 지불 후 제거 |
| 건강 검진 | 이번 턴 전역 통행료 절반 |
| 극심한 분리불안 | 가장 먼 플레이어 위치로 즉시 도착 |
| 신의뜻 | 같은 칸 말 조우 시 조각 기반 돈 갈취 버프 |
| 도움 닫기 | 조우 시 추가 이동 버프 |
| 우대권 | 이번 턴 통행료 1회 면제 |
| 무료 증정 | 이번 턴 토지 1회 무료 구매 |
| 가벼운 분리불안 | 같은 칸 말 조우 시 현금 획득 버프 |
| 마당발 | 같은 색 인접 토지 추가 구매 1회 |
| 강제 매각 | 일반 잔꾀 단계에선 보류; 도착 타이밍용 |
| 호객꾼 | 일반 잔꾀 단계에선 보류; 말 장애물 성격 |
| 뇌고왕 | 이번 턴 본인 통행료 절반 |
| 재뿌리기 | 지정 토지 통행료 0 |
| 긴장감 조성 | 지정 토지 통행료 2배 이상 |
| 느슨함 혐오자 | 이번 턴 전역 통행료 2배 |
| 극도의 느슨함 혐오자 | 영구 전역 통행료 2배 |
| 무역의 선물 | 자기 토지 1개와 상대 토지 1개 교환 |
| 뭘리권 | 일반 잔꾀 단계에선 보류; 주사위 재굴림용 |
| 뭔칙휜 | 일반 잔꾀 단계에선 보류; 강화 재굴림용 |
| 과속 | 2냥 지불, 주사위 +1 |
| 저속 | 2냥 획득, 주사위 -1 |
| 뇌절왕 | 자기 토지 구역 재도착 시 연쇄 이동 버프 |
| 이럇! | 생존 플레이어 전원 주사위 +1 |
| 번뜩임 | 잔꾀 교환 |
| 아주 큰 화목 난로 | 조각 +1, F +1 |
| 거대한 산불 | 조각 +2, F +2 |

## 상태 플래그로 처리되는 카드들

실제 게임 진행 중에는 아래 같은 상태 플래그가 세팅됩니다.

- `trick_free_purchase_this_turn`
- `trick_dice_delta_this_turn`
- `trick_personal_rent_half_this_turn`
- `trick_same_tile_cash2_this_turn`
- `trick_same_tile_shard_rake_this_turn`
- `trick_one_extra_adjacent_buy_this_turn`
- `trick_encounter_boost_this_turn`
- `trick_force_sale_landing_this_turn`
- `trick_zone_chain_this_turn`

즉 잔꾀는 전부 즉시 결과만 내는 것이 아니라, 많은 카드가 “이번 턴 상태 플래그”를 켜고 이후 이동/구매/착지 해석에 영향을 줍니다.

## 현재 구조의 중요한 특징

### 1. 카드 텍스트가 곧 메타데이터는 아니다
- 현재 `언제나` 여부는 설명 문자열에 의존합니다.
- 장기적으로는 `timing`, `effect_family`, `visibility`, `requires_window` 같은 구조화 메타가 필요합니다.

### 2. 일반 잔꾀 단계에서 안 쓰는 카드가 있다
- 모든 잔꾀가 같은 시점에 소비되는 것이 아닙니다.
- `강제 매각`, `뭘리권`, `뭔칙휜`, `호객꾼`은 별도 타이밍 성격이 강합니다.

### 3. 잔꾀는 현재 1차 시각화 구현에서 제외하는 것이 맞다
이유:
- 카드 자체 정의
- 공개/비공개 처리
- 즉시 발동 / 상태 플래그 / 보류형 카드
- 보급 burden 교환
- 언제나 사용 가능 카드의 별도 타이밍
이 한 묶음이어서, 다른 시스템보다 UI/입력 계약이 더 복잡합니다.

따라서 잔꾀는 먼저 소스 기준 정리와 수동 점검을 한 뒤, 별도 파이프라인으로 분리하는 것이 안전합니다.
